# State Pattern

## 배경
- 동작이 내부 상태에 따라 달라지는 시스템에서 조건을 통해 상태 변경을 관리하면 코드가 복잡해지고 유지 관리가 어려워짐. 
  - if 문이 복잡해지게되는 상황

## 개념
- 무언가의 '상태'가 각각 그에 해당하는 클래스의 객체로 존재함.
- 각 클래스가 메소드를 각각 구현

## 구조
- Context: 상태에 따라 동작이 달라지는 클래스
- State Interface: 다양한 상태에 대한 동작을 정의하는 인터페이스
- Concrete State Classes: State 인터페이스를 구현하고 특정 상태에 대한 동작을 정의하는 클래스

## 쉽게 이해하기
- 자판기를 떠올리면 됨
  - "동전 없음", "동전 투입됨", "음료 배출 중" 등 자판기의 **상태마다 버튼을 눌렀을 때 동작이 다름**
  - if/else로 상태를 분기하는 대신, 각 상태를 별도 클래스로 만들어서 해당 상태의 동작을 직접 구현
  - 상태가 바뀌면 Context가 들고 있는 State 객체만 교체하면 됨
- Strategy Pattern과 비슷하게 보이지만, **State는 상태 간 전환이 핵심**이고 Strategy는 알고리즘 교체가 핵심
