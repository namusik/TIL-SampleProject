# 이미지 포맷 (JPEG · PNG · WebP · AVIF)

> 최종 업데이트: 2026-06-10 | JPEG ISO/IEC 10918, PNG ISO/IEC 15948, WebP VP8L/VP8X 기준

## 개념

이미지 포맷은 **픽셀 데이터를 파일로 저장하는 방식(압축 알고리즘 + 메타데이터 구조)** 을 정의한다. "어떤 정보를 얼마나 버릴 것인가"에 따라 손실 압축(Lossy)과 무손실 압축(Lossless)으로 나뉘며, 이 선택이 화질·파일 크기·투명도 지원 여부를 결정한다.

> 비유: 악보를 저장하는 방법. 모든 쉼표·다이나믹까지 완벽히 기록(PNG)하거나, 사람이 듣기 어려운 고음부 세부 정보는 생략해서 파일 크기를 줄인다(JPEG). 무엇을 버릴지가 포맷의 설계 철학이다.

**JPG와 JPEG는 완전히 동일한 포맷**이다. MS-DOS/Windows 95의 파일명 3글자 제한 때문에 `.jpeg` → `.jpg`로 줄인 것이 굳어진 것이며, 파일 내용·바이너리 구조 모두 같다.

## 배경/역사

| 연도 | 사건 |
|------|------|
| 1986 | ISO/IEC JTC 1이 정지 화상 압축 표준화 작업 시작, **JPEG 위원회(Joint Photographic Experts Group)** 발족 |
| 1992 | JPEG 표준 확정 (ISO/IEC 10918-1). 사진 저장 용량을 기존 대비 10:1로 줄임 |
| 1995 | GIF의 LZW 알고리즘 특허 문제(Unisys)로 대안 필요 → **PNG(Portable Network Graphics)** 개발 시작 |
| 1996 | PNG 1.0 공개. W3C 권고안으로 채택 |
| 2010 | Google이 VP8 코덱 기반 **WebP** 공개. JPEG 대비 25~35% 작은 파일 크기 |
| 2019 | Alliance for Open Media(AOM)가 AV1 인트라 프레임 기반 **AVIF** 1.0 공개 |
| 2020 | Chrome·Firefox가 AVIF 지원 시작, 브라우저 지원 본격화 |
| 2023 | Safari 16+, iOS 16+에서 AVIF 정식 지원. WebP·AVIF가 사실상 웹 표준으로 정착 |

## 포맷 비교

| 항목 | JPEG | PNG | WebP | AVIF |
|------|------|-----|------|------|
| **표준** | ISO/IEC 10918 | ISO/IEC 15948 | Google/IETF RFC 6386 | AOM AV1 |
| **압축 방식** | 손실 (Lossy) | 무손실 (Lossless) | 손실 + 무손실 | 손실 + 무손실 |
| **알고리즘** | DCT + 양자화 | DEFLATE (zlib) | VP8/VP8L | AV1 인트라 프레임 |
| **투명도(알파)** | ❌ | ✅ 8비트 알파 | ✅ | ✅ |
| **애니메이션** | ❌ | ✅ APNG | ✅ | ✅ |
| **색 깊이** | 8비트 채널 | 8/16비트 채널 | 8비트 | 10/12비트 HDR |
| **파일 크기** | 기준 | 크다 (2~5배) | 작다 (~30%) | 더 작다 (~50%) |
| **브라우저 지원** | 전부 | 전부 | 전부 | Safari 16+, Chrome 85+, FF 93+ |
| **인코딩 속도** | 빠름 | 빠름 | 보통 | 느림 |

## 손실 vs 무손실

### 손실 압축 (JPEG)

JPEG는 **DCT(이산 코사인 변환)** 로 이미지를 주파수 성분으로 분해한 뒤, 사람 눈이 둔감한 고주파(미세한 질감·경계) 성분을 버린다.

- 장점: 파일 크기가 매우 작음. 사진처럼 부드러운 색조 변화에 최적
- 단점: **저장할 때마다 품질 손상 누적**. 글자·선화 경계에 링잉(Ringing) 노이즈 발생

```
원본 → JPEG 저장 → 열기 → 편집 → JPEG 저장 → ... (반복)
                  ↑ 매번 품질 손실 누적 (되돌릴 수 없음)
```

품질 설정(Quality 0~100)은 양자화 테이블의 세밀도를 제어한다. Quality 85 이상은 육안 구분이 거의 불가능하며, 웹 용도 기준으로 75~85가 일반적.

### 무손실 압축 (PNG)

PNG는 **DEFLATE(zlib)** 알고리즘으로 픽셀 데이터를 압축한다. 픽셀 값이 1비트도 변하지 않는다.

- 장점: 화질 손실 없음. 알파 채널(투명도) 지원. 다이어그램·스크린샷·로고에 적합
- 단점: 파일 크기가 사진에는 불리함. JPEG 대비 2~5배 큰 경우가 많음

## 언제 무엇을 쓰나

| 상황 | 권장 포맷 | 이유 |
|------|----------|------|
| 사진 (웹 게시) | WebP > JPEG | 파일 크기 절감. IE 지원 불필요하면 WebP 우선 |
| 로고·아이콘 (래스터) | PNG | 날카로운 경계, 투명 배경 필요 |
| 스크린샷·UI 목업 | PNG | 텍스트 경계 선명, 무손실 필요 |
| 텍스트 포함 이미지 | PNG | JPEG는 글자 주변에 링잉 노이즈 |
| 애니메이션 이미지 | WebP > APNG > GIF | GIF는 256색 제한 |
| HDR 사진·스트리밍 | AVIF | 10비트 색 깊이, 최고 압축률 |
| 로고·일러스트 (벡터) | SVG | 해상도 독립적, 확대해도 선명 |
| 원본 편집 보관 | PNG 또는 TIFF | 손실 없이 반복 편집 가능 |

## JPEG 링잉 노이즈

```
"Hello" 텍스트 저장 시:

  PNG → H e l l o  (선명, 픽셀 그대로)
 JPEG → H e l l o  (글자 주변에 번짐·아티팩트 발생)
```

JPEG의 8×8 픽셀 블록 처리 특성상, 급격한 명도 변화(텍스트 경계, 선화)에서 특히 두드러진다. 품질을 95 이상으로 높이면 줄어들지만 파일 크기 이점도 함께 줄어든다.

## JPEG 블록 아티팩트 (Block Artifact)

JPEG가 이미지를 8×8 블록 단위로 처리하기 때문에, 저화질(Quality 50 이하)에서는 이미지가 격자 형태로 구획되어 보이는 현상이 발생한다. 이는 JPEG 고유의 대표적인 열화 패턴이다.

## 차세대 포맷 (WebP · AVIF)

### WebP

- Google이 2010년 VP8 비디오 코덱을 정지 화상에 적용해 만든 포맷
- 손실 모드: JPEG 대비 같은 화질에 약 25~35% 작은 파일
- 무손실 모드: PNG 대비 약 26% 작은 파일
- 투명도·애니메이션 모두 지원
- 현재 모든 주요 브라우저 지원 (IE 제외)

```html
<!-- picture 태그로 WebP 우선, JPEG 폴백 -->
<picture>
  <source srcset="photo.webp" type="image/webp">
  <img src="photo.jpg" alt="사진">
</picture>
```

### AVIF

- AV1 비디오 코덱의 인트라 프레임만 추출한 정지 화상 포맷
- WebP 대비 10~20% 추가 절감 (같은 화질 기준)
- 10비트·12비트 HDR 지원
- 단점: **인코딩 속도가 느림**. 빌드/CDN 파이프라인에서 사전 인코딩 필요
- Safari 16(2022), Chrome 85(2020), Firefox 93(2021) 이상에서 지원

## GIF는 왜 쓰지 않나

| 항목 | GIF | 현대 대안 |
|------|-----|----------|
| 색상 수 | 256색 (8비트 팔레트) | WebP·APNG: 풀컬러 |
| 파일 크기 | 크다 | 동급 화질 대비 5~10배 차이 |
| 투명도 | 1비트 (완전 불투명/완전 투명만) | PNG·WebP: 8비트 알파 |
| 용도 | 레거시 애니메이션 | MP4(H.264) 또는 WebM이 대부분 대체 |

짧은 루프 애니메이션도 MP4/WebM이 GIF 대비 파일 크기가 훨씬 작다. GitHub·Slack 등 대부분의 플랫폼이 업로드 GIF를 내부적으로 MP4로 변환한다.

## 래스터 vs 벡터

위에서 다룬 모든 포맷(JPEG·PNG·WebP·AVIF·GIF)은 **래스터(Raster)** 이미지 — 픽셀의 격자로 이미지를 표현한다. 확대하면 픽셀이 보인다.

**SVG(Scalable Vector Graphics)** 는 **벡터** 포맷 — XML로 도형·경로의 수식을 저장한다. 어떤 크기로 렌더링해도 항상 선명하다. 로고·아이콘·지도처럼 기하학적 도형으로 구성된 이미지에 적합하다.
