# 프로토콜의 구성 요소

> 최종 업데이트: 2026-04-22 | 기준: 네트워크 프로토콜 공통 개념 정리

## 개념

새로운 네트워크 프로토콜(HTTP, WebSocket, SSE, gRPC 등)을 공부하면 **공통으로 등장하는 어휘와 개념**이 있다. 이 문서는 그 용어들을 한곳에 모아 설명한다. **프로토콜을 이해할 때 "어떤 관점들을 보면 되는지"** 를 정리한 기반 문서.

> 비유하자면 모든 요리책이 공통으로 다루는 "재료·조리법·도구·플레이팅" 같은 기본 항목들. 프로토콜마다 내용은 달라도, **짚어봐야 할 축**은 같다.

## 프로토콜을 볼 때의 8가지 관점

어떤 프로토콜이든 아래 8가지를 각각 확인하면 대부분 파악된다.

| # | 관점 | 핵심 질문 |
|---|------|---------|
| 1 | **표준 문서** | 누가·어디서 정의했나? |
| 2 | **와이어 포맷** | 네트워크에 어떤 바이트가 흐르나? |
| 3 | **MIME / Content-Type** | 어떤 Content-Type으로 식별되나? |
| 4 | **메시지 구조** | 헤더·바디·프레임이 어떻게 생겼나? |
| 5 | **핸드셰이크/연결 관리** | 어떻게 연결하고 끊나? |
| 6 | **인코딩** | 데이터를 어떻게 변환하나? |
| 7 | **시맨틱스** | 값·명령의 의미 규칙은? |
| 8 | **보안(TLS)** | 어떻게 암호화하나? |

---

## 1. 표준 문서 (Specification)

프로토콜은 **표준화 기구**가 발행한 문서로 정의된다. 구현자들이 이 문서를 따라 서로 호환되는 소프트웨어를 만든다.

### 주요 표준화 기구

| 기구 | 성격 | 대표 표준 |
|------|------|---------|
| **IETF** (Internet Engineering Task Force) | 인터넷 표준 | HTTP, WebSocket, QUIC, TLS — **RFC 번호**로 식별 |
| **WHATWG** | 웹 생명 표준 | HTML, Fetch, SSE (`EventSource`) — Living Standard |
| **W3C** | 웹 표준 | DOM, WebRTC, ARIA |
| **IEEE** | 하드웨어/저수준 | Wi-Fi(802.11), Ethernet(802.3) |
| **ISO/OSI** | 모델·참조 | OSI 7계층 모델 |
| **벤더 자체** | 단독 발표 | Apple Push, AWS Signature |

### RFC 보는 법

- **RFC = Request For Comments**, IETF 표준 문서
- RFC 번호는 **갱신 시 새 번호** 부여 (예: HTTP/1.1이 RFC 2616 → 7230 → 9110)
- 상태: Proposed Standard → Internet Standard → Historic
- 예: `RFC 9110 HTTP Semantics`, `RFC 6455 The WebSocket Protocol`

## 2. 와이어 포맷 (Wire Format)

**네트워크 선(wire) 위에 실제로 흐르는 바이트의 모양**. 프로토콜의 "물리적 실체"이자 가장 구체적인 부분.

### 텍스트 vs 바이너리

| 구분 | 특징 | 예 |
|------|------|----|
| **텍스트** | 사람이 읽을 수 있음, 디버깅 쉬움, 파싱 비용↑ | HTTP/1.x, SMTP, STOMP, SSE |
| **바이너리** | 컴팩트·빠름, 디버깅 어려움, 툴 필요 | HTTP/2, WebSocket 프레임, gRPC, TCP |

### 예시

**HTTP/1.1 (텍스트)**
```
GET /index.html HTTP/1.1\r\n
Host: example.com\r\n
\r\n
```

**WebSocket 프레임 (바이너리)**
```
0x81 0x05 0x48 0x65 0x6C 0x6C 0x6F
│    │    └─────"Hello"─────────┘
FIN/op  len
```

### 관련 용어

- **Framing** — 메시지를 프레임(작은 단위)으로 쪼개는 방식
- **Chunked Transfer** — HTTP에서 응답을 나눠 보내는 방식
- **Stream** — 논리적으로 이어지는 바이트 흐름

## 3. MIME / Content-Type

**데이터가 어떤 종류인지 나타내는 표준 식별자**. HTTP 헤더 `Content-Type` 값으로 전달된다.

### 구조

```
대분류(type) / 소분류(subtype) ; 파라미터

application/json
text/html; charset=utf-8
multipart/form-data; boundary=----abc
```

### 대분류(Type)

| Type | 용도 | 예 |
|------|------|----|
| `text/*` | 텍스트 | `text/html`, `text/plain`, `text/event-stream` |
| `application/*` | 구조화된 바이너리/텍스트 | `application/json`, `application/pdf`, `application/octet-stream` |
| `image/*` | 이미지 | `image/png`, `image/jpeg`, `image/webp` |
| `audio/*` / `video/*` | 미디어 | `audio/mp3`, `video/mp4` |
| `multipart/*` | 여러 파트 묶음 | `multipart/form-data`, `multipart/mixed` |

### 자주 쓰는 MIME

| MIME | 용도 |
|------|------|
| `application/json` | **REST API 표준** |
| `application/x-www-form-urlencoded` | HTML 폼 기본 |
| `multipart/form-data` | 파일 업로드 |
| `text/html` | HTML 문서 |
| `text/event-stream` | **SSE** |
| `application/grpc` | gRPC |
| `application/octet-stream` | 일반 바이너리 |

### "전용 MIME"이 있다는 의미

어떤 프로토콜이 **자기만의 표준 Content-Type을 갖는다** = "이 Content-Type을 보면 이 프로토콜이라고 식별할 수 있다"
- SSE는 `text/event-stream` 전용
- gRPC는 `application/grpc` 전용
- 반면 Polling/Long Polling은 전용 MIME 없음 — 그냥 `application/json` 등 일반 HTTP

## 4. 메시지 구조 (Message Structure)

**하나의 메시지가 어떤 필드들로 구성되는가**. 보통 "헤더 + 바디" 또는 "프레임 + 페이로드"의 조합.

### 예시

**HTTP 메시지**
```
[시작 라인]    GET /users HTTP/1.1
[헤더들]       Host: ...
              Authorization: ...
[빈 줄]
[바디]         {"query": "..."}
```

**WebSocket 프레임**
```
[FIN][opcode][MASK][payload length][masking key][payload]
```

**SSE 이벤트**
```
id: 1
event: update
data: {"msg": "hello"}
          ← 빈 줄이 이벤트 경계
```

### 공통 구성 요소

| 구성 | 역할 |
|------|------|
| **헤더/메타** | 라우팅·인증·타입 정보 |
| **바디/페이로드** | 실제 데이터 |
| **경계** (Delimiter) | 메시지 구분 (`\r\n\r\n`, `\n\n`, boundary 등) |
| **크기 정보** (Length) | 파싱에 필수 (`Content-Length`, `Payload len`) |

## 5. 핸드셰이크와 연결 관리

**연결을 어떻게 맺고 유지하고 끊는가**.

### 핸드셰이크

- **TCP 3-way handshake** — SYN → SYN-ACK → ACK (모든 TCP 기반의 공통)
- **TLS handshake** — 암호화 키 교환
- **WebSocket handshake** — HTTP `Upgrade: websocket` → `101 Switching Protocols`
- **HTTP/2 preface** — `PRI * HTTP/2.0\r\n` 시작 시퀀스

### 연결 관리 개념

| 용어 | 의미 |
|------|------|
| **Keep-Alive** | 연결을 재사용해 여러 요청을 주고받음 |
| **Persistent Connection** | 장기 유지되는 연결 |
| **Pipelining** | 응답을 기다리지 않고 여러 요청 전송 |
| **Multiplexing** | 하나의 연결에 여러 스트림을 섞어 보냄 (HTTP/2) |
| **Heartbeat / Ping-Pong** | 연결 생존 확인을 위한 주기적 메시지 |
| **Graceful Close** | 정상 종료 절차 (WebSocket의 CLOSE 프레임 등) |

## 6. 인코딩 (Encoding)

**데이터를 네트워크에 맞게 변환하는 방식**. 크게 **문자 인코딩**과 **압축**, **특수 인코딩**으로 나뉨.

### 문자 인코딩

| 인코딩 | 설명 |
|-------|------|
| **UTF-8** | 전 세계 문자 표현, 인터넷 표준 |
| UTF-16 | 일부 내부 표현 (Java String 등) |
| ASCII | 영문 7비트, UTF-8의 부분집합 |
| EUC-KR, CP949 | 한글 레거시 |

### 압축

| 알고리즘 | 용도 |
|---------|------|
| **gzip** | HTTP 응답 압축 기본 |
| **br (Brotli)** | gzip보다 좋은 압축률 |
| **deflate** | zlib 기반 |
| **Snappy**, **LZ4** | 속도 최적화 (Kafka 등) |

### 특수 인코딩

| 인코딩 | 용도 |
|-------|------|
| **base64** | 바이너리를 텍스트로 (이메일 첨부, multipart) |
| **URL 인코딩** | URL에 못 쓰는 문자 변환 (`%20` 등) |
| **HPACK** | HTTP/2 헤더 압축 |
| **QPACK** | HTTP/3 헤더 압축 |
| **Chunked** | Content-Length 없이 조각조각 전송 |

## 7. 시맨틱스 (Semantics)

**값의 의미 규칙**. 와이어 포맷(모양)과 달리, "이 값이 어떤 행동을 유발해야 하는지" 정의.

### HTTP 예시

- **메서드 시맨틱스**: `GET`은 조회(안전), `PUT`은 멱등, `POST`는 생성 등
- **상태 코드 시맨틱스**: `2xx` 성공, `4xx` 클라이언트 오류, `5xx` 서버 오류
- **캐싱 시맨틱스**: `Cache-Control`, `ETag`, `Last-Modified`

### 일반 개념

| 용어 | 의미 |
|------|------|
| **Safe** | 서버 상태를 변경하지 않음 (GET) |
| **Idempotent** | 여러 번 호출해도 결과 동일 (PUT, DELETE) |
| **Cacheable** | 응답을 캐시 가능 |
| **Stateless** | 요청이 이전 요청에 의존하지 않음 |

→ 같은 와이어 포맷(HTTP 메시지)이라도 **메서드와 코드의 시맨틱스를 지키느냐**가 REST의 핵심.

## 8. 보안 (TLS)

현대 프로토콜은 거의 모두 **TLS 위에서 동작**. 평문 통신은 개발·내부망 외에는 사실상 사라짐.

| 프로토콜 | 평문 포트 | TLS 포트 |
|---------|---------|---------|
| HTTP / HTTPS | 80 | 443 |
| WebSocket (ws / wss) | 80 | 443 |
| SMTP | 25 | 465/587 (SMTPS/STARTTLS) |
| MQTT | 1883 | 8883 |

### TLS 관련 용어

- **핸드셰이크** — 대칭 키 교환. TLS 1.3부터 1-RTT, 0-RTT(재연결) 가능
- **인증서(Certificate)** — 서버 신원 증명 (X.509)
- **SNI (Server Name Indication)** — 한 IP의 여러 도메인 분기
- **ALPN** — 핸드셰이크 중 "HTTP/1.1인가 HTTP/2인가" 협상
- **mTLS** — 클라이언트도 인증서 제시

## 프로토콜 분석 템플릿

새로운 프로토콜을 볼 때 이 표를 채워보면 이해가 빠르다.

| 관점 | 예: HTTP/1.1 | 예: WebSocket | 예: SSE |
|------|-------------|---------------|--------|
| 표준 | RFC 9110 | RFC 6455 | WHATWG HTML |
| 와이어 포맷 | 텍스트 | 바이너리 프레임 | 텍스트 (HTTP 응답) |
| 전용 MIME | — | — | `text/event-stream` |
| 메시지 구조 | 시작라인+헤더+바디 | FIN/opcode/mask/payload | `id:/event:/data:` |
| 연결 | Keep-Alive | 단일 지속 | 단일 지속 HTTP 응답 |
| 인코딩 | gzip, UTF-8 | 마스킹(클라→서버) | UTF-8 |
| 시맨틱스 | 메서드/상태코드 | opcode 종류 | 이벤트 타입 |
| 보안 | HTTPS (443) | wss:// (443) | HTTPS (443) |

## 요약

- 프로토콜은 **"표준 문서가 규정한 와이어 포맷 + 시맨틱스 + 연결 관리 규약"** 의 집합
- 새 프로토콜을 볼 때는 이 문서의 **8가지 관점**으로 쪼개서 보면 된다
- "자체 와이어 포맷이 있나", "전용 MIME이 있나" 같은 질문이 **프로토콜인지 기법인지 구분하는 기준**이 됨
- 개별 프로토콜 문서(`HTTP.md`, `WebSocket.md` 등)에서 이 용어들이 나오면 이 문서를 참고
