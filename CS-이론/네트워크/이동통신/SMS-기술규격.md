# SMS 기술 규격

> 최종 업데이트: 2026-06-11 | 3GPP TS 23.040 / TS 23.038 기준

## 개념

**SMS(Short Message Service, 단문 메시지 서비스)** 는 이동통신망에서 **짧은 텍스트 메시지를 주고받는 서비스**다. 3GPP 표준(TS 23.040, TS 23.038)이 메시지 구조와 인코딩을 정의한다.

> 비유: 음성 통화를 위해 깔아둔 '신호용 골목길(시그널링 채널)'에 편지 한 장을 슬쩍 끼워 보내는 것. 통화용 대역을 따로 점유하지 않고 제어 신호 틈에 실어 보내기 때문에, 메시지 크기가 **태생적으로 작게** 제한된다.

SMS의 모든 제약(140바이트, 한글 70자 등)은 이 "신호 채널에 얹어 보낸다"는 출발점에서 나온다. 표준 규격은 [3GPP](3GPP.md)가 관장한다.

## 배경/역사

| 연도 | 사건 |
|------|------|
| 1985 | GSM 표준화 과정에서 SMS 개념 명세에 포함 |
| 1992 | 세계 최초 SMS 발송 — 영국, "Merry Christmas" (PC→휴대폰) |
| 1990년대 후반 | 피처폰 보급과 함께 폭발적 성장. 통신사 핵심 수익원 |
| 2000년대 | 한국에서 **LMS·MMS**(장문·멀티미디어) 도입 |
| 2010년대 | 카카오톡 등 OTT 메신저에 일상 대화 영역 잠식 |
| 현재 | 인증번호(OTP)·알림 등 **B2C 자동 발송** 용도로 견고하게 잔존. **RCS**(차세대 문자)로 진화 시도 |

SMS는 음성 통화용으로 설계된 GSM 망의 **제어 채널(signaling channel)** 여유 공간을 활용하려는 아이디어에서 출발했다. "이미 깔린 신호 통로에 짧은 글을 끼워 보내자"는 발상이라, 처음부터 대용량을 의도하지 않았다.

## 메시지 구조 (TS 23.040)

SMS 메시지는 **PDU(Protocol Data Unit)** 형식으로 인코딩되어 전송된다. PDU는 여러 필드로 구성되는데, 그중 **본문(User Data) 필드 하나의 최대 크기가 140바이트(1,120비트)** 다. 이 140바이트가 SMS의 글자 수 제한을 결정한다.

핵심은 **140바이트는 순수 본문(User Data)만의 한도**라는 점이다. 발신번호·수신번호·타임스탬프·인코딩 종류는 **140바이트 바깥의 별도 헤더 필드**이며, 본문 140바이트를 갉아먹지 않는다.

| PDU 필드 | 내용 | 140바이트에 포함? |
|----------|------|:---:|
| TP-OA | 발신번호 | ❌ 별도 헤더 |
| TP-DA | 수신번호 | ❌ 별도 헤더 |
| TP-SCTS | 타임스탬프 | ❌ 별도 헤더 |
| TP-DCS | 인코딩 종류 | ❌ 별도 헤더 |
| **TP-UD** | **본문 (User Data)** | ✅ **이 필드가 최대 140바이트** |

그래서 뒤에 나올 영문 160자(7비트 × 160 = 140바이트)·한글 70자(2바이트 × 70 = 140바이트)가 **순수 본문 기준**으로 정확히 떨어진다. 헤더까지 합친 값이었다면 이 계산이 맞지 않는다.

### PDU — 정해진 형식으로 직렬화한다는 뜻

**PDU(Protocol Data Unit)** 는 프로토콜이 "데이터를 이런 순서·구조로 담아라"고 규정한 **한 덩어리의 형식**이다. 보통 **헤더(제어 정보) + 페이로드(본문)** 로 구성된다. SMS를 "PDU 형식으로 인코딩한다"는 말은, 사람이 읽는 메시지를 **위 필드들이 정해진 순서대로 바이트(16진수)로 직렬화**한다는 뜻이다.

> 비유: **택배 송장**. 받는사람·보내는사람·내용물 칸이 정해진 순서로 있고, 그 양식에 맞춰 적어야 배송 시스템이 읽는다. SMS PDU도 "어느 자리에 발신번호, 어느 자리에 인코딩 타입, 그 뒤가 본문"이 규격으로 박혀 있다.

예를 들어 "hello"를 SMS PDU로 인코딩하면 다음과 같은 16진수 문자열이 된다.

```
0011000B915121551532F40000AA05E8329BFD06
└SMSC┘└수신번호┘        └DCS┘  └본문(hello)┘
```

사람 눈엔 의미 없는 16진수 덩어리지만, 단말기·SMSC는 규격을 알기 때문에 **"3번째가 인코딩 타입, 그 뒤가 본문"** 하고 정확히 잘라 읽는다. 이렇게 모든 필드를 16진수로 직렬화하는 방식을 **PDU 모드**라 하며, 한글(UCS-2)·분할 메시지 같은 모든 기능을 쓰려면 이 모드를 사용한다. (사람이 읽는 형태로 간단히 주고받는 **텍스트 모드**도 있으나 인코딩·옵션 제어가 제한적이다.)

## SMS 전송 흐름 — 직렬화부터 역직렬화까지

SMS 한 통이 발신 폰에서 수신 폰까지 가는 과정은 **"발신 측 직렬화 → 무선·코어망 전송 → 수신 측 역직렬화"** 의 대칭 구조다.

```
[발신 폰]                                          [수신 폰]
"안녕"                                              "안녕"
  │ ① PDU로 직렬화(바이트화)                          ▲ ⑥ 역직렬화 → 화면 표시
  ▼                                                  │
[기지국] ──② 코어망──▶ [SMSC] ──④ 수신자 위치 조회──▶ [기지국]
          (발신측)      ③ 저장      (HLR/HSS)         (수신측) ⑤ 무선으로 내려줌
```

| 단계 | 무슨 일이 일어나나 |
|------|------------------|
| ① 직렬화 | 발신 폰이 "안녕"을 **PDU(바이트/16진수)로 인코딩** |
| ② 무선 전송 | 가까운 **기지국**으로 PDU가 무선으로 올라감 → 코어망으로 라우팅 |
| ③ 저장 | **SMSC**가 일단 보관 (store) |
| ④ 위치 조회 | 가입자 위치 DB(HLR/HSS)에 "수신자가 지금 어느 기지국?" 질의 |
| ⑤ 무선 전송 | 수신자 쪽 기지국으로 PDU를 내려보냄 → 수신 폰으로 (forward) |
| ⑥ 역직렬화 | 수신 폰이 PDU를 **파싱·디코딩해서 화면에 표시** |

### SMSC — 보관 후 전달 (Store-and-Forward)

발신자와 수신자는 통화처럼 **직접 연결을 맺지 않는다(connectionless)**. 그래서 중간에 **SMSC(SMS Center, 단문 메시지 센터)** 가 끼어들어, 메시지를 **일단 저장했다가 수신자가 망에 접속하면 전달**한다. 이메일 서버와 똑같은 "보관 후 전달" 방식이라, 전원이 꺼진 상대에게 보내도 켜는 순간 도착한다. 발신자는 SMSC까지만 던지면 끝이고(fire-and-forget), 수신자가 켜져 있는지는 신경 쓰지 않는다.

### 무선 구간은 전파로 한 겹 더 변조된다

PDU(바이트)는 핸드폰↔기지국 사이에서 다시 **전파(RF 신호)로 변조**되어 날아간다. 둘 사이는 전선이 아니라 공기이기 때문이다. 다만 **논리적으로는 PDU 바이트가 그대로 전달**되는 것으로 보면 된다 — "바이트를 전파에 실어 보내고, 받아서 다시 바이트로 복원"하는 셈이다.

### 역직렬화 = 발신의 정확한 역순

수신 폰이 하는 일은 발신의 거울이다. 핵심은 **본문을 디코딩하기 전에 인코딩 종류(DCS)부터 읽는다**는 점이다.

| 발신 (직렬화) | 수신 (역직렬화) |
|--------------|----------------|
| "안녕" → 인코딩 타입(DCS) 결정 | PDU에서 **DCS 필드 읽어 인코딩 판별** |
| 본문을 UCS-2/7비트로 바이트화 | 그 인코딩으로 **본문 바이트를 글자로 디코딩** |
| 필드 순서대로 PDU 조립 | 필드를 순서대로 **잘라냄(파싱)** → 화면 렌더링 |

여기서 [한글-인코딩.md](../../한글-인코딩.md)의 원리가 그대로 적용된다. 수신 폰이 **DCS(인코딩 종류)를 먼저 봐야** 본문 바이트를 올바른 표로 디코딩한다. 발신이 UCS-2로 보냈는데 수신이 엉뚱한 표로 읽으면 **글자 깨짐(Mojibake)** 이 발생한다. PDU에 인코딩 타입을 박아두는 이유가 바로 이것이다.

## 문자 인코딩 (TS 23.038)

140바이트 안에 **몇 글자**를 담을 수 있는지는 인코딩 방식에 따라 갈린다. TS 23.038은 세 가지를 규정한다.

| 인코딩 | 글자당 크기 | 1건 최대 글자 수 | 용도 |
|--------|-----------|----------------|------|
| **GSM 7비트** | 7비트 | **160자** | 영문·숫자·기본 기호 |
| 8비트 | 8비트 | 140자 | 바이너리 데이터 |
| **UCS-2 (UTF-16)** | 16비트(2바이트) | **70자** | 한글 등 비라틴 문자 |

핵심은 **영문은 7비트로 욱여넣어 160자**까지 되지만, **한글은 UCS-2 2바이트라 70자**로 줄어든다는 점이다. 한 메시지에 한글이 한 글자라도 섞이면 전체가 UCS-2로 전환되어 70자 기준이 적용된다.

### 한글과 EUC-KR — 왜 통신사 문자는 EUC-KR 계열인가

SMS는 140바이트라는 빡빡한 예산 안에서 동작하므로, **한글 한 글자의 바이트가 적을수록 유리**하다.

| 인코딩 | 한글 1자 | 140바이트에 담기는 한글 |
|--------|---------|----------------------|
| **EUC-KR / CP949** | **2바이트** | 약 70자 |
| UCS-2 (UTF-16) | 2바이트 | 70자 |
| UTF-8 | 3바이트 | 약 46자 |

UTF-8(한글 3바이트)을 쓰면 같은 140바이트에 46자밖에 못 담는다. 반면 EUC-KR/UCS-2(2바이트)는 70자가 들어간다. 이 바이트 효율 + 수십 년 굳어진 망 표준 때문에, 웹은 UTF-8로 전환됐어도 **통신사 문자(SMS) 계열은 여전히 EUC-KR/2바이트 인코딩**을 쓴다. → [한글-인코딩.md](../../한글-인코딩.md)

## 긴 메시지 — Concatenated SMS

70자(한글)·160자(영문)를 넘기면 어떻게 될까. SMS는 **여러 건으로 쪼개 보내고 수신 단말이 다시 합치는** 방식(Concatenated SMS, 분할 SMS)을 쓴다.

- 140바이트 중 일부를 **UDH(User Data Header)** 헤더로 떼어내, "이건 3장 중 2번째" 같은 조립 정보를 담는다
- 그만큼 본문 공간이 줄어 **분할 시 한글은 67자, 영문은 153자**로 감소
- 사용자에겐 한 통으로 보이지만, 망에서는 여러 건으로 과금된다

```
"긴 한글 메시지..." (한글 200자)
  → [1/3] 67자 + [2/3] 67자 + [3/3] 66자  ← 3건으로 분할 전송 → 단말이 재조립
```

## SMS vs LMS vs MMS (한국 통신사 분류)

140바이트 제약을 넘는 수요 때문에, 한국 통신사는 길이·형식별로 등급을 나눈다.

| 구분 | 정식 명칭 | 용량 | 내용 |
|------|----------|------|------|
| **SMS** | Short Message Service | 한글 약 **40~45자(80~90바이트)** | 단문 텍스트 |
| **LMS** | Long Message Service | 한글 **1,000자(2,000바이트)** | 장문 텍스트 |
| **MMS** | Multimedia Message Service | 2,000바이트 + 첨부 | 이미지·동영상 포함 |

### SMS 90바이트는 '표준'이 아니라 '사업자 운영 기준'

한국 통신사 SMS의 실사용 한도는 흔히 **90바이트(한글 45자)** 로 알려져 있다. 다만 이건 3GPP가 정한 표준값이 **아니라 각 사업자가 정하는 서비스 운영 기준**이라, 한 장짜리 통합 공식 규격서가 따로 있는 게 아니다. 실제로 안내 기준이 사업자마다 갈린다.

| 사업자/채널 | SMS 한도 |
|------------|---------|
| LG U+·다수 문자 중계사 | 한글 **45자 (90바이트)** |
| KT | 한글 **40자 (80바이트)** |

**90바이트(45자)가 가장 널리 통용되는 기준선**이고, KT처럼 40자로 더 보수적으로 안내하는 곳도 있다고 이해하면 된다.

이 값이 표준 본문 한도(140바이트, 한글 70자)보다 좁은 이유는 **헤더 때문이 아니다.** 앞서 봤듯 발신번호·타임스탬프 같은 헤더는 140바이트 바깥의 별도 필드라 본문을 갉아먹지 않는다. 90바이트는 **사업자가 SMS라는 상품 등급을 한글 45자(2바이트 × 45 = 90바이트)로 그어놓은 과금 경계선**일 뿐이다. 초창기 피처폰 시절 한글 40~45자로 출발한 관행이 굳어진 것이고, **이 한도를 넘으면 LMS로 자동 전환**되어 과금 단가가 오른다.

즉 **표준의 물리적 한계(140바이트)와 사업자의 상품 한계(80~90바이트)는 다른 층위**다. 140은 3GPP가 정한 본문 최대치, 80~90은 통신사가 그은 SMS/LMS 과금 경계선이다. LMS/MMS는 SMS와 전송 방식 자체가 다르다(멀티미디어 채널 사용).

## 백엔드에서 SMS 발송

서버에서 SMS를 직접 통신망에 쏘지는 않는다. 보통 **문자 중계사(SMS 게이트웨이)** 의 API를 호출하고, 중계사가 SMSC로 전달한다.

```
백엔드 서버 → 문자 중계사 API(HTTP) → SMSC → 이동통신망 → 수신 단말
```

- 국내: NHN Cloud(Toast), 알리고, 네이버 클라우드 SENS, 카카오 알림톡 등
- 글로벌: Twilio, AWS SNS 등
- 발송 시 **90바이트(45자)를 넘으면 자동으로 LMS로 전환**되어 과금 단가가 올라가므로, 인증번호·알림 문구는 길이를 의식해 작성한다

## 정리

- **SMS** = 이동통신 제어 채널에 얹어 보내는 단문 서비스. 표준은 [3GPP](3GPP.md) TS 23.040/23.038
- **전송 흐름**: 발신 폰 PDU 직렬화 → 기지국·코어망 → SMSC 보관후전달 → 수신 폰 DCS 판별 후 역직렬화
- **140바이트** 페이로드 → 영문 160자 / 한글 70자(UCS-2)
- 한글은 2바이트 인코딩이 유리 → 통신사 문자가 EUC-KR 계열을 쓰는 근거
- 길면 **분할 전송(Concatenated SMS)**, 더 길면 **LMS/MMS** 로 등급 전환
- 백엔드는 **문자 중계사 API** 를 거쳐 발송
