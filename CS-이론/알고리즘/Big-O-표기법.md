# Big-O 표기법 (시간 복잡도)

> 최종 업데이트: 2026-05-17 | CS 이론 보편 개념 기준 (특정 언어 비종속)

## 개념
**입력 크기 n이 커질 때 알고리즘의 실행 시간(또는 메모리)이 얼마나 빠르게 늘어나는지를 나타내는 표기법.** 핵심은 "정확한 실행 시간"이 아니라 **증가 추세(성장률)** 다.

요리에 비유하면, "이 레시피는 30분 걸린다"가 아니라 **"손님이 2배로 늘면 시간도 2배로 늘어난다"** 처럼 *규모가 커질 때 어떻게 변하는가*를 말하는 것이다. 컴퓨터 성능·언어·상수는 환경마다 다르므로, 그런 세부는 무시하고 **n에 대한 변화 양상만** 본다.

그래서 Big-O는 두 가지를 버린다:

| 버리는 것 | 예시 | 이유 |
|-----------|------|------|
| 상수 계수 | `3n` → `O(n)` | n이 커지면 계수 3은 의미가 작아짐 |
| 낮은 차수 항 | `n² + n` → `O(n²)` | n이 커지면 n²이 전체를 지배 |

즉 Big-O는 **n이 충분히 클 때의 상한(upper bound), 최악의 경우 성장률**을 표현한다.

## 배경/역사
Big-O는 컴퓨터과학에서 만든 표기가 아니라 **수학(정수론)에서 빌려온 것**이다.

- **1894** — 독일 수학자 **파울 바흐만(Paul Bachmann)** 이 저서 *Analytische Zahlentheorie*에서 "Ordnung(차수, order of)"의 의미로 기호 `O`를 처음 도입
- **1909** — **에드문트 란다우(Edmund Landau)** 가 확장 → 둘의 이름을 따 **바흐만–란다우 표기법(Bachmann–Landau notation)** 이라 부름
- **1950년대** — 응용수학의 점근 분석(asymptotic analysis)에 사용
- **1970년대** — **도널드 커누스(Donald Knuth)** 가 알고리즘 분석에 도입·대중화하면서 오늘날 CS의 표준 도구가 됨 (Knuth는 1976년 이 기호를 대문자 오미크론으로 해석)

## 자주 쓰이는 복잡도 등급
n이 커질수록 느려지는 순서대로:

| 표기 | 이름 | 대표 예시 | n이 10배 되면 |
|------|------|-----------|---------------|
| **O(1)** | 상수 | 배열 인덱스 접근, 해시 조회 | 변화 없음 |
| **O(log n)** | 로그 | 이진 탐색, 균형 트리 탐색 | 거의 안 늘어남 |
| **O(n)** | 선형 | 배열 전체 순회 | 10배 |
| **O(n log n)** | 선형로그 | 병합 정렬, 퀵 정렬(평균) | 약 10배+ |
| **O(n²)** | 제곱 | 이중 루프, 버블 정렬 | 100배 |
| **O(2ⁿ)** | 지수 | 부분집합 완전 탐색 | 사실상 폭발 |
| **O(n!)** | 팩토리얼 | 외판원 완전 탐색 | 사실상 계산 불가 |

비유로 감을 잡으면:
- **O(1)** — 책의 'p.150'을 바로 펼치기
- **O(log n)** — 사전에서 단어 찾기(절반씩 좁히기)
- **O(n)** — 1쪽부터 한 장씩 넘기며 찾기
- **O(n²)** — 모든 사람이 다른 모든 사람과 한 번씩 악수

## 코드로 보는 복잡도
```text
// O(1) — 입력 크기와 무관하게 한 번
return arr[0]
```
```text
// O(n) — 루프가 n번
for i in 0..n:
    sum += arr[i]
```
```text
// O(n²) — 루프 안에 루프
for i in 0..n:
    for j in 0..n:
        check(i, j)
```
```text
// O(log n) — 매번 범위를 절반으로
while lo <= hi:
    mid = (lo + hi) / 2
    ...
```

## 최선·평균·최악 (Big-O / Big-Ω / Big-Θ)
같은 알고리즘도 입력에 따라 속도가 다르다. 그래서 경계가 셋이다.

| 표기 | 의미 | 보장하는 것 |
|------|------|-------------|
| **O (Big-O)** | 상한 | "아무리 나빠도 이보다 느리진 않다" (최악) |
| **Ω (Big-Omega)** | 하한 | "아무리 좋아도 이보다 빠르진 않다" (최선) |
| **Θ (Big-Theta)** | 상·하한 일치 | 최선=최악 차수가 같을 때 (정확한 차수) |

실무에서 "복잡도"라 하면 보통 **최악의 경우 = Big-O**를 말한다. 최악을 알면 성능을 보장할 수 있기 때문이다.

## 시간 복잡도 vs 공간 복잡도
Big-O는 **시간**뿐 아니라 **메모리 사용량**에도 똑같이 적용된다.

- **시간 복잡도** — 연산 횟수가 n에 따라 어떻게 증가하는가
- **공간 복잡도** — 추가로 쓰는 메모리가 n에 따라 어떻게 증가하는가 (입력 자체 크기는 제외)

둘은 종종 **트레이드오프** 관계다. 예를 들어 결과를 미리 배열에 저장해 두면 조회는 O(1)로 빨라지지만 O(n) 메모리를 더 쓴다 — 캐싱·메모이제이션의 본질이 바로 이 교환이다.

## 주의: 점근 표기의 한계
Big-O는 **n이 충분히 클 때**의 이야기다. 따라서:

- 입력이 작으면 O(n²) 알고리즘이 O(n log n)보다 실제로 빠를 수 있다 (상수가 작아서). 실제로 다수의 표준 라이브러리 정렬은 작은 구간에서 삽입 정렬로 전환한다.
- 상수와 하위 항을 버리므로 "O(n)이면 무조건 빠르다"는 보장이 아니다 — 같은 O(n)이라도 상수 차이로 수 배 날 수 있다.

즉 Big-O는 **알고리즘 선택의 1차 기준**이지, 측정(profiling)을 대체하지는 않는다.
