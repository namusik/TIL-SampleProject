# 한글 인코딩

> 최종 업데이트: 2026-06-10 | KS X 1001, Unicode 15.1 기준

## 개념

한글 인코딩은 **한글 문자를 바이트로 저장·전송하기 위한 규칙 체계**다. 컴퓨터는 바이트만 다루므로, '가'라는 문자도 반드시 특정 규칙(CES, Character Encoding Scheme)에 따라 바이트로 변환해야 한다. 어떤 규칙을 쓰느냐에 따라 EUC-KR, CP949, UTF-8 등이 달라진다.

> 비유: 한글을 전보로 보내는 상황. 한국전신공사 규칙표(EUC-KR)로 찍은 신호를 국제전신 규칙표(UTF-8)로 읽으면 전혀 다른 내용이 된다. 규칙표(인코딩 방식)가 달라서 생기는 문제다.

인코딩·디코딩·CCS·CES 개념은 [Character-Encoding.md](Character-Encoding.md)에서 다룬다. 이 문서는 **한글 인코딩의 역사와 방식**에 집중한다.

## 배경/역사

| 연도 | 사건 |
|------|------|
| 1974 | 한국 정부 최초 한글 코드 표준안 논의 시작. 조합형·완성형 방식 대립 |
| 1987 | **KS C 5601** 제정 (현 KS X 1001). **완성형** 채택, 한글 **2,350자** 수록 |
| 1989 | **아래아한글(HWP)** 출시. 당시 정부 표준(완성형)이 아닌 **조합형** 방식 채택 |
| 1991 | MBC 만화 **'똠방각하'** 방영. '똠' 자가 KS C 5601에 없어 전산 처리 불가 → 완성형 한계 대중에 부각 |
| 1992 | 정보통신부, 조합형 병행 표준안 추진했으나 완성형 단일 표준 유지 |
| 1995 | Microsoft가 Windows 95 한국어판에 **CP949(확장 완성형)** 도입. EUC-KR의 상위 호환으로 한글 **11,172자** 전부 지원 |
| 1996 | **Unicode 2.0**에 한글 11,172자 정식 수록 (Hangul Syllables: U+AC00~U+D7A3) |
| 2000년대 | 인터넷 보급으로 EUC-KR이 한국 웹 사실상 표준. 브라우저·서버 기본값으로 자리잡음 |
| 2010년대 | 스마트폰·글로벌 서비스 확산으로 UTF-8 전환 가속. HTML5가 UTF-8을 권장 인코딩으로 명시 |
| 현재 | 신규 시스템은 사실상 UTF-8 단일화. EUC-KR·CP949는 레거시 시스템에만 잔존 |

## ASCII와 한글의 충돌

ASCII는 7비트 128자(0x00~0x7F)로 영문만 표현한다. 1바이트 중 상위 128자(0x80~0xFF)는 ASCII 미정의 영역이었고, 각국은 이 영역을 자국어로 채웠다.

- **유럽**: ISO-8859 시리즈 (Latin 계열)
- **한국**: EUC-KR (KS C 5601 기반)
- **일본**: Shift_JIS, EUC-JP
- **중국**: GB2312, Big5

한글은 ASCII 1바이트로 표현 불가능하므로 **2바이트** 조합으로 표현했다. 이것이 멀티바이트 인코딩의 시작이다.

## 조합형

한글의 제자(製字) 원리 — 초성·중성·종성 — 에 각각 코드를 부여해 조합하는 방식이다.

- 초성(19자) + 중성(21자) + 종성(27자 + 없음) = 이론상 **11,172자** 전부 표현 가능
- 2바이트 조합형: 각 바이트의 최상위 비트(MSB)를 1로 세팅해 ASCII와 구분

```
'가' = 초성 ㄱ(코드) + 중성 ㅏ(코드) + 종성 없음(코드)
'힣' = 초성 ㅎ(코드) + 중성 ㅣ(코드) + 종성 ㅎ(코드)
```

| 장점 | 단점 |
|------|------|
| 11,172자 전부 표현 | 당시 구현 복잡, 처리 비용 |
| 한글 제자 원리에 충실 | 완성형 기반 시스템과 호환 불가 |
| 아래아한글이 채택 | 정부 표준(완성형)에서 배제 |

## 완성형

완성된 글자 단위(가, 나, 힣 등)에 코드를 직접 부여하는 방식이다. 1987년 KS C 5601로 표준화됐다.

- **KS C 5601 (현 KS X 1001)**: 자주 쓰이는 한글 **2,350자** 선정·수록
- 구현이 단순하고 ASCII 기반 시스템과 연동 용이

**문제**: 2,350자에 없는 글자는 표현 자체가 불가능했다.

```
표현 불가 예시 (KS C 5601 기준):
  똠, 뚝, 뗄, 솩, 괜, 뇨 ...
```

1991년 MBC 만화 '똠방각하'의 '똠' 자가 전산에서 처리되지 않자 사회적 논란이 됐다. 방송 자막·TV 편성표·신문 프로그램 안내에서 '똠'을 표시할 방법이 없었다.

## EUC-KR

Extended Unix Code-KR. KS C 5601 완성형 한글 2,350자를 Unix 환경에서 사용할 수 있도록 정의한 인코딩이다.

- 한글: 2바이트 (0xA1A1~0xFEFE 범위)
- ASCII: 1바이트 (0x00~0x7F)
- 한글 이외 문자(한자, 특수문자 등) 포함

```
'가' → B0 A1
'힣' → C7 D1 (수록된 2,350자 중 마지막 글자)
'똠' → 표현 불가 ❌
```

## CP949 (확장 완성형, MS949)

Microsoft가 1995년 Windows 95 한국어판에 도입한 EUC-KR의 **상위 호환** 확장이다.

- EUC-KR의 2,350자를 그대로 포함 + 한글 8,822자 추가 = **11,172자 전부 지원**
- 기존 EUC-KR 텍스트를 CP949로 읽어도 동일하게 표시 (상위 호환)
- Windows 한국어판 기본 인코딩. `java.nio.charset.Charset.forName("EUC-KR")`이 실제로는 CP949를 반환

```java
Charset.forName("EUC-KR").displayName();  // "EUC-KR" (실제 구현은 CP949)
"똠".getBytes("EUC-KR").length;           // 2 (CP949로 처리되어 정상 인코딩)
```

| 항목 | EUC-KR | CP949 |
|------|--------|-------|
| 수록 한글 | 2,350자 | 11,172자 |
| 표준 | KS C 5601 | Microsoft 독자 확장 |
| Windows 기본 | ❌ | ✅ |
| 상위 호환 | — | EUC-KR 포함 |

## 유니코드에서의 한글

유니코드는 한글을 두 가지 방식으로 수록한다.

### 한글 음절 (Hangul Syllables)

- 범위: **U+AC00 ~ U+D7A3** (11,172자)
- 완성형 글자 11,172자를 모두 코드 포인트로 직접 수록
- '가'(U+AC00)부터 '힣'(U+D7A3)까지 순서대로 배치

```
'가' = U+AC00
'나' = U+B098
'힣' = U+D7A3
```

### 한글 자모 (Hangul Jamo)

- 범위: **U+1100 ~ U+11FF** (초성·중성·종성 개별 코드)
- 조합형 방식. 자모를 조합해 음절을 구성
- 현대 한글 외에 고어(古語) 자모도 수록

```
ㄱ = U+1100 (초성)
ㅏ = U+1161 (중성)
→ 조합: U+1100 U+1161 = '가' (렌더링 시 합자)
```

유니코드 정규화(NFC/NFD)에서 음절 방식과 자모 조합 방식이 혼재할 수 있어, 한글 문자열 비교 시 정규화가 필요한 이유가 된다.

## 인코딩 방식에 따라 바이트가 달라진다

같은 글자라도 **어떤 인코딩(CES)을 쓰느냐에 따라 바이트 수도, 실제 바이트 값도 둘 다 달라진다.** '문자 → 바이트'는 인코딩표를 거쳐야 정해지는 변환이므로, 표가 바뀌면 결과 바이트 배열이 통째로 바뀐다.

같은 코드 포인트라도 인코딩이 다르면 다른 바이트가 나온다. 예를 들어 '가'는 CCS(유니코드)에서 코드 포인트 `U+AC00`을 부여받지만, 이 코드 포인트를 바이트로 바꾸는 규칙(CES)이 UTF-8이냐 EUC-KR이냐에 따라 결과물이 달라진다.

| 인코딩 | '가' 바이트 | 크기 |
|--------|-----------|------|
| UTF-8 | EA B0 80 | 3바이트 |
| UTF-16 BE | AC 00 | 2바이트 |
| EUC-KR | B0 A1 | 2바이트 |
| CP949 | B0 A1 | 2바이트 (EUC-KR 범위 내 동일) |

한글 '한' 글자로도 동일하게 확인된다.

| 인코딩 | '한' 바이트 | 크기 |
|--------|-----------|------|
| UTF-8 | ED 95 9C | 3바이트 |
| UTF-16 BE | D5 5C | 2바이트 |
| EUC-KR / CP949 | C7 D1 | 2바이트 |

### 한글은 보통 EUC-KR이 더 짧다

- **EUC-KR**: '한글+한자+영문' 정도만 담는 좁은 표라 한글을 **2바이트**에 욱여넣는다
- **UTF-8**: 전 세계 문자를 담는 가변길이 인코딩이라, 한글 대역(U+AC00~U+D7A3)은 **3바이트** 구간에 떨어진다

그래서 한글이 많은 문서는 EUC-KR·UTF-16이 UTF-8보다 파일 크기가 작을 수 있다. 다만 웹·네트워크 표준은 UTF-8이므로 전송 시 변환이 발생한다.

### 단, ASCII 문자는 인코딩이 달라도 동일하다

영문·숫자 같은 ASCII 문자는 EUC-KR이든 UTF-8이든 **똑같이 1바이트**다. UTF-8이 ASCII와 호환되도록 설계됐기 때문이다.

| 글자 | UTF-8 | EUC-KR | 비고 |
|------|-------|--------|------|
| `A` | 41 | 41 | 동일 (1바이트) |
| `1` | 31 | 31 | 동일 (1바이트) |
| `가` | EA B0 80 | B0 A1 | 다름 (3 vs 2바이트) |

즉 "UTF-8이 항상 더 크다"가 아니라, **한글 같은 비ASCII 문자에서만 UTF-8이 더 크다.** 그리고 EUC-KR로 만든 바이트 배열을 UTF-8로 읽으면 깨지는 근본 이유도 이것이다 — 바이트 배열 자체가 서로 다른 규칙으로 만들어졌기 때문이다.

## Java에서의 한글 인코딩

- Java의 `String`은 내부적으로 **UTF-16**으로 문자열을 저장한다
  - Java 9+: **Compact Strings** 도입 — Latin-1 범위(ASCII)만 있는 문자열은 1바이트/문자로 저장해 메모리 절약. 한글이 포함되면 UTF-16 유지
- 문자열 입출력 시에만 지정한 인코딩(UTF-8, EUC-KR 등)으로 변환한다
- JVM 기본 인코딩(`file.encoding`)은 JVM 로딩 시에만 초기화되므로, 코드 중간에 프로퍼티를 변경해도 반영되지 않는다

```java
String s = "가";

// 인코딩별 바이트 확인
byte[] utf8  = s.getBytes(StandardCharsets.UTF_8);  // [EA, B0, 80] — 3바이트
byte[] utf16 = s.getBytes(StandardCharsets.UTF_16); // [AC, 00] — 2바이트 (BOM 제외)
byte[] euckr = s.getBytes("EUC-KR");                // [B0, A1] — 2바이트

// EUC-KR 바이트를 UTF-8로 읽으면 깨짐
String broken = new String(euckr, StandardCharsets.UTF_8);  // '??'
String correct = new String(euckr, "EUC-KR");               // '가'
```

Spring Boot에서 레거시 EUC-KR 클라이언트 대응:

```java
@Bean
public FilterRegistrationBean<CharacterEncodingFilter> encodingFilter() {
    CharacterEncodingFilter filter = new CharacterEncodingFilter();
    filter.setEncoding("UTF-8");
    filter.setForceEncoding(true);
    return new FilterRegistrationBean<>(filter);
}
```
