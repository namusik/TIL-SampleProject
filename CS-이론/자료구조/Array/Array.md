# Array 배열

> 최종 업데이트: 2026-05-17 | CS 이론 보편 개념 기준 (특정 언어 비종속)

## 개념
**동일한 타입의 데이터를 연속된 메모리 공간에 일렬로 저장하는 자료구조.** 각 데이터는 0부터 시작하는 정수 **인덱스(index)** 로 식별한다.

아파트 우편함을 떠올리면 쉽다. 우편함들이 벽에 **빈틈없이 나란히** 붙어 있고, 각 칸에는 호수(인덱스)가 적혀 있다. "102호"라고 하면 입구에서 몇 걸음 떨어진 곳인지 곧바로 계산되므로, 처음 칸부터 하나씩 세지 않아도 바로 찾아갈 수 있다. 이 "바로 찾아간다"가 배열의 핵심 성질이다.

배열을 정의하는 세 가지 특성:

| 특성 | 의미 |
|------|------|
| 연속성(contiguous) | 모든 원소가 메모리에 끊김 없이 붙어서 저장됨 |
| 동질성(homogeneous) | 모든 원소가 같은 타입 → 원소 크기가 일정 |
| 고정 크기(fixed size) | 생성 시 크기가 결정됨 (순수 배열 기준) |

## 배경/역사
배열은 **가장 오래된 자료구조 중 하나**다. 1940년대 폰 노이만 구조(von Neumann architecture)에서 메모리가 "주소가 매겨진 연속된 칸의 나열"로 설계되면서, 그 물리적 구조를 그대로 추상화한 것이 배열이다. 즉 배열은 하드웨어 메모리 모델과 **개념이 거의 1:1로 대응**하는 가장 원초적인 자료구조이며, 그래서 거의 모든 언어가 가장 기본 타입으로 내장하고 있다. 이후 등장한 동적 배열·리스트·해시 테이블 등 상당수 자료구조가 내부적으로 배열을 토대로 구현된다.

## 메모리 구조와 임의 접근 (Random Access)
배열의 가장 큰 강점인 **O(1) 인덱스 접근**은 메모리 주소 계산에서 나온다.

```text
원소 주소 = 시작 주소(base) + (인덱스 × 원소 크기)
```

예: `int`(4바이트) 배열의 시작 주소가 1000일 때

| 인덱스 | 계산 | 주소 |
|--------|------|------|
| arr[0] | 1000 + 0×4 | 1000 |
| arr[1] | 1000 + 1×4 | 1004 |
| arr[3] | 1000 + 3×4 | 1012 |

여기서 **원소 크기**란 한 칸이 메모리에서 차지하는 바이트 수(`int`=4, `double`=8 등)다. 이 값은 **타입에 의해 결정**되고 그 타입은 배열 생성 시점에 고정되므로, 원소 크기 역시 그때 확정되어 변하지 않는다. 모든 칸의 크기가 같다는 **동질성**이 보장되기 때문에 비로소 `인덱스 × 원소 크기`라는 곱셈이 성립한다 — 즉 동질성은 단순 제약이 아니라 O(1) 임의 접근을 가능하게 하는 핵심 전제다. (동적 배열이 재할당돼도 타입은 그대로라 원소 크기는 불변이며, 바뀌는 것은 시작 주소와 용량뿐이다.)

인덱스만 알면 곱셈·덧셈 한 번으로 주소가 즉시 나온다. 앞에서부터 순회할 필요가 없으므로 **어떤 위치든 동일한 시간(O(1))** 에 접근 — 이것을 **임의 접근(random access)** 이라 한다. 만약 칸마다 타입(크기)이 제각각이라면 이 곱셈이 통하지 않아 앞에서부터 누적해야 하므로 O(n)이 된다. 연결 리스트(Linked List)가 못 하는 바로 그 일이다.

## 연산별 시간 복잡도
| 연산 | 시간 복잡도 | 이유 |
|------|------------|------|
| 인덱스 접근 `arr[i]` | **O(1)** | 주소 계산 한 번 |
| 값 검색 (정렬 안 됨) | O(n) | 전부 훑어야 함 |
| 맨 끝 삽입/삭제 | O(1) | 다른 원소 영향 없음 |
| 중간/앞 삽입 | **O(n)** | 뒤 원소 전부 한 칸씩 밀어야 함 |
| 중간/앞 삭제 | **O(n)** | 빈 칸을 메우려 앞으로 당겨야 함 |

"줄 서 있는 사람들 사이에 새치기로 끼어들면, 뒷사람이 전부 한 발씩 물러나야 한다" — 이것이 중간 삽입이 O(n)인 이유다. **빠른 조회 / 느린 중간 삽입·삭제**가 배열의 트레이드오프다.

## 정적 배열 vs 동적 배열
순수 배열은 크기가 고정이라 불편하다. 그래서 대부분의 언어는 **동적 배열(dynamic array)** 을 함께 제공한다 (Java `ArrayList`, C++ `vector`, Python `list`, Go `slice`).

| 구분 | 정적 배열 (static) | 동적 배열 (dynamic) |
|------|-------------------|---------------------|
| 크기 | 생성 시 고정 | 자동 확장 |
| 내부 구조 | 그 자체 | **내부에 배열을 품고 있음** |
| 임의 접근 | O(1) | O(1) (동일) |
| 끝에 추가 | 불가(꽉 차면) | **분할 상환 O(1)** |

**"내부에 배열을 품고 있음"의 의미**: 동적 배열은 새로운 자료구조가 아니라, 평범한 고정 배열을 멤버로 감싼 **래퍼(wrapper)** 다. 그래서 두 개의 수를 따로 관리한다.

| 필드 | 의미 |
|------|------|
| `size` | 실제로 채워진 원소 개수 (사용자가 보는 길이) |
| `capacity` | 내부 고정 배열의 진짜 크기 (여유 칸 포함) |

`capacity`가 `size`보다 크게 여유 칸을 미리 잡아두므로, 평소 추가는 빈 칸에 넣기만 하면 돼 O(1)이다.

동적 배열의 확장 원리: 내부 배열이 꽉 차면(`size == capacity`) 보통 **2배 크기의 새 배열을 만들어 전체를 복사**하고 기존 것을 버린다. 큰 집으로 이사하는 것과 같다 — 짐(데이터)이 방을 꽉 채우면 더 큰 집을 구해 전부 옮기고 헌 집을 비운다. 매번이 아니라 꽉 찼을 때만 이사하므로, 복사 순간만 O(n)이지만 그 한 번의 비용을 그 사이의 수많은 값싼 추가 연산에 나눠 보면 평균 O(1)이 된다 — 이를 **분할 상환 분석(amortized analysis)** 이라 한다. 내부가 결국 배열이므로 인덱스 접근은 그대로 O(1)이고, 중간 삽입·삭제는 똑같이 O(n)이며, `capacity > size` 여유분만큼 메모리를 약간 더 쓴다.

```text
용량 4 → 가득 참 → 용량 8 새 배열 할당 → 4개 복사 → 이후 4번은 복사 없이 추가
```

## 다차원 배열과 메모리 배치 순서
2차원 이상 배열은 논리적으로는 격자지만, 물리 메모리는 1차원이라 **펼치는 순서**가 필요하다.

| 방식 | 펼치는 순서 | 사용 언어 |
|------|------------|-----------|
| Row-major | 행 우선 (한 행을 다 채우고 다음 행) | C, Java, Python |
| Column-major | 열 우선 (한 열을 다 채우고 다음 열) | Fortran, MATLAB, R |

이게 실무에서 중요한 이유: 메모리에 붙어 있는 순서대로 순회해야 **CPU 캐시 적중률**이 높다. Row-major 언어에서 2중 루프를 `[열][행]` 순으로 돌리면 메모리를 건너뛰며 접근해 같은 코드라도 수 배 느려질 수 있다(캐시 미스).

## 배열 vs 연결 리스트
둘 다 "여러 값을 순서대로 담는다"는 목적은 같지만 메모리 전략이 정반대다.

| 항목 | 배열 | 연결 리스트 |
|------|------|------------|
| 메모리 배치 | 연속 | 흩어짐 (노드마다 포인터로 연결) |
| 임의 접근 | O(1) | O(n) (앞에서부터 따라가야 함) |
| 중간 삽입/삭제 | O(n) (밀기) | O(1) (포인터만 교체, 위치를 알 때) |
| 추가 메모리 | 없음 | 노드마다 포인터 |
| 캐시 효율 | 높음 | 낮음 |

조회가 많고 구조 변경이 적으면 배열, 중간 삽입·삭제가 잦으면 연결 리스트가 유리하다.
