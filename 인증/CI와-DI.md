# CI와 DI (연계정보와 중복가입확인정보)

> 최종 업데이트: 2026-04-17 | 대한민국 본인확인서비스 기준

## 개념

CI와 DI는 **주민등록번호를 직접 저장하지 않고도 본인을 확인**할 수 있도록 만들어진 대체 식별값이다.

온라인 서비스에서 본인인증(휴대폰 인증, 아이핀 등)을 수행하면, 본인확인기관(NICE, KMC 등)이 주민등록번호를 기반으로 CI와 DI를 생성해서 서비스 사업자에게 전달한다.

> 비유하자면, CI는 전국 어디서든 통하는 **여권 번호** 같은 것이고, DI는 특정 건물에서만 쓰는 **출입카드 번호** 같은 것이다.

## CI (Connecting Information, 연계정보)

- 주민등록번호를 **단방향 암호화(해시)**하여 생성한 **88byte** 값
- 어떤 서비스에서 인증하든 **동일인이면 항상 같은 CI**가 나옴
- 서로 다른 서비스 간에 동일인인지 연계·확인할 때 사용

> 마치 주민등록번호의 "암호화 분신"처럼, 서비스가 달라도 같은 사람이면 같은 값이 나온다.

### 활용 사례

- 금융권 ↔ 통신사 간 동일인 확인
- 여러 서비스에 걸친 본인 계정 통합
- 법적 요청에 따른 서비스 간 이용자 대조

## DI (Duplication Information, 중복가입확인정보)

- 주민등록번호 + **사업자 고유 코드**를 조합하여 해시한 **64byte** 값
- 같은 사람이라도 **서비스마다 다른 DI**가 나옴
- 한 서비스 내에서 동일인이 여러 계정을 만드는 것을 방지할 때 사용

> 같은 사람이어도 건물(서비스)이 바뀌면 출입카드 번호가 달라지는 것과 같다.

### 활용 사례

- 이벤트 중복 참여 방지
- 동일 서비스 내 다중 계정 탐지
- 탈퇴 후 재가입 제한

## CI vs DI 비교

| 구분 | CI (연계정보) | DI (중복가입확인정보) |
|------|--------------|-------------------|
| 길이 | 88byte | 64byte |
| 동일인 + 같은 사이트 | 동일 | 동일 |
| 동일인 + 다른 사이트 | **동일** | **다름** |
| 생성 입력값 | 주민등록번호 | 주민등록번호 + 사업자 코드 |
| 주요 용도 | 서비스 간 본인 연계 | 서비스 내 중복가입 방지 |
| 개인정보 민감도 | 상대적으로 **높음** (서비스 간 추적 가능) | 상대적으로 **낮음** (서비스 범위 한정) |

## 본인인증 흐름과 CI/DI 발급

```
사용자 → 서비스(웹/앱) → 본인확인기관(NICE, KMC 등)
                              │
                     휴대폰 인증 / 아이핀 등
                              │
                     주민등록번호 기반 해시 생성
                              │
                    ┌─────────┴─────────┐
                    │                   │
                   CI 생성            DI 생성
              (주민번호 해시)    (주민번호+사업자코드 해시)
                    │                   │
                    └─────────┬─────────┘
                              │
                      서비스에 CI, DI 전달
                      (주민번호는 전달 안 함)
```

## 법적 근거

- **정보통신망법** 및 **본인확인기관 지정 제도**에 의해 운영
- 2012년 주민등록번호 수집 금지 이후, CI/DI가 주민등록번호의 대체 식별 수단으로 자리잡음
- CI는 개인정보에 해당하므로 수집·이용 시 **개인정보 처리 동의**가 필요

## 백엔드 개발 시 고려사항

| 항목 | 설명 |
|------|------|
| 저장 | CI/DI 모두 개인정보이므로 **암호화 저장** 필수 (DB 컬럼 암호화) |
| 인덱싱 | 조회 빈도가 높으면 암호화된 값 자체를 인덱스로 활용 |
| 본인확인기관 연동 | NICE, KMC 등의 API 연동 시 CI/DI를 응답으로 수신 |
| CI 기반 회원 매칭 | 여러 서비스를 운영하는 경우 CI로 동일 회원 식별 가능 |
| DI 기반 중복 체크 | 회원가입 시 DI 중복 여부를 체크하여 다중 가입 방지 |
| 개인정보 파기 | 회원 탈퇴 시 CI/DI도 함께 파기하거나 분리 보관 정책 적용 |
