# 프롬프트 엔지니어링 (Prompt Engineering)
> 최종 업데이트: 2026-05-02 | 기준 정보: Claude 4.x, GPT-4o, 멀티모달 모델 시점

## 개념
프롬프트 엔지니어링은 **LLM에게 보내는 입력(프롬프트)을 정교하게 작성하여 원하는 결과를 얻어내는 기술**이다. 한 번의 요청에서 모델 동작을 최대한 끌어내는 데 초점이 있다.

쉽게 비유하면, 같은 신입 직원에게 *"이거 해줘"* 와 *"이런 배경 상황에서, 이런 형식으로, 이 예시처럼 해줘"* 를 시키면 결과 품질이 완전히 달라진다. LLM도 마찬가지로 **입력의 정교함이 출력 품질을 결정**한다.

## 배경/역사
- **2020-05**: OpenAI GPT-3 논문에서 **"in-context learning"** 능력 발견. 별도 학습 없이 예시 몇 개만 보여줘도 패턴을 따라 함 (Few-shot learning)
- **2022-01**: Google Brain의 ["Chain-of-Thought Prompting" 논문 (Wei et al.)](https://arxiv.org/abs/2201.11903). *"단계별로 생각해봐"* 라는 한 줄 추가만으로 수학·추론 정확도 대폭 향상 발견. 프롬프트 자체가 모델 능력을 *끌어낸다* 는 인식 확산
- **2022-11**: ChatGPT 출시. 일반 대중이 LLM과 대화하면서 *어떻게 묻는지가 결과를 좌우한다* 는 점이 부각
- **2023**: "프롬프트 엔지니어" 가 정식 직무로 등장. 일부 회사가 별도 채용 공고를 냄. 관련 책·강의 폭발적 출시
- **2024**: 모델이 정교화되면서 단순 프롬프트만으론 한계 → **컨텍스트 엔지니어링** 이라는 더 큰 개념으로 확장
- **2026-02**: Mitchell Hashimoto의 "Engineer the Harness" 글 이후 **하네스 엔지니어링** 이 프롬프트/컨텍스트의 더 바깥 영역으로 자리잡음
- **현재(2026)**: "프롬프트 엔지니어 직무는 죽었다" 담론과 *여전히 핵심 스킬* 이라는 의견이 공존. 독립 직무보다 **모든 엔지니어의 기본 소양** 으로 흡수되는 경향

## 컨텍스트 엔지니어링과의 차이

| 구분 | 프롬프트 엔지니어링 | 컨텍스트 엔지니어링 |
|------|---------------------|---------------------|
| 단위 | 한 번의 요청(프롬프트) | 워크플로우/에이전트 루프 전체 |
| 관심사 | "어떻게 잘 물어볼까?" | "무엇을 / 얼마나 컨텍스트에 둘까?" |
| 대상 | 텍스트 표현, 예시(Few-shot) | 토큰 예산 분배, 컨텍스트 구조 |
| 비용 관점 | 보통 무관 | 핵심 (토큰 = 비용) |
| 사용 시점 | 1회성 작업 | 장기 실행 에이전트, 챗봇, RAG |

→ **프롬프트 엔지니어링 ⊂ 컨텍스트 엔지니어링 ⊂ 하네스 엔지니어링** 의 입문 단계.

## 핵심 기법

### 1. Zero-shot vs Few-shot
- **Zero-shot**: 예시 없이 지시만
- **Few-shot**: 예시 몇 개를 같이 줘서 패턴을 학습시킴

```
[Zero-shot]
"다음 문장을 긍정/부정으로 분류해라: 영화가 정말 재밌었다"

[Few-shot]
"문장: 영화가 별로다 → 부정
 문장: 음식이 맛있다 → 긍정
 문장: 영화가 정말 재밌었다 → ?"
```

### 2. Chain of Thought (CoT, 사고의 사슬)
복잡한 추론 문제에 *"단계별로 생각해봐"* 한 줄을 추가하면 정확도가 크게 오른다.

```
일반:    "23 × 47 = ?"
CoT:    "23 × 47 = ? 단계별로 계산해줘."
        → 23 × 40 = 920
          23 × 7  = 161
          합계    = 1081
```

### 3. Role Prompting (역할 부여)
*"너는 ~다"* 로 역할을 정의하면 응답 톤·관점이 일관됨.

```
"너는 보안에 민감한 시니어 백엔드 엔지니어다.
다음 코드를 리뷰해라."
```

### 4. 출력 형식 명시
모호한 출력 대신 구체 형식을 지정.

```
"결과를 JSON으로 답해. 스키마: { name: string, age: number }"
"비교 결과를 마크다운 표로 정리해."
```

### 5. XML 태그 구조화 (Claude 특히 강함)
Claude는 학습 시 XML 데이터를 많이 봤기 때문에 태그 인식률이 높다.

```xml
<role>코드 리뷰어</role>
<rules>
- 보안 이슈를 우선 지적
- 한국어로 응답
</rules>
<code>
... 첨부 코드 ...
</code>
<task>위 코드를 리뷰해라.</task>
```

### 6. Multi-shot Examples (좋은 예 + 나쁜 예)
*"이렇게 답해"* 와 *"이렇게 답하지 마"* 를 둘 다 보여주면 더 정확해진다.

### 7. Prefilling (응답 미리 채우기)
응답의 시작 부분을 미리 채워서 출력 형식을 강제.

```python
# Claude API
messages=[
    {"role": "user", "content": "JSON 형식으로 답해줘"},
    {"role": "assistant", "content": "{"}  # 모델이 이 뒤를 이어서 작성
]
```

### 8. 고급 기법 (간략 소개)
| 기법 | 핵심 아이디어 |
|------|---------------|
| Self-consistency | 같은 질문을 N번 던져 다수결 |
| Tree of Thought (ToT) | 가능한 답 트리를 가지치기로 탐색 |
| ReAct | Reasoning + Acting — 추론과 도구 호출을 번갈아 |
| Reflexion | 자기 답을 비평한 뒤 재작성 |

## Anthropic 공식 권장 사항 (Claude 기준)
[Anthropic Prompt Engineering 가이드](https://docs.claude.com/en/docs/build-with-claude/prompt-engineering) 요약.

1. **명확하고 직접적으로** — 모호한 지시보다 구체적 동사·기준 사용
2. **예시 사용 (Multi-shot)** — 5개 이상의 다양한 예시가 효과적
3. **Claude에게 생각할 시간 주기** — `<thinking>` 태그 또는 단계별 지시
4. **XML 태그로 구조화** — `<role>`, `<example>`, `<documents>`, `<task>` 등
5. **System 프롬프트 적극 활용** — 역할·규칙·맥락은 system에, 작업은 user에
6. **프리필링 활용** — 출력 형식이 까다로울수록 효과적
7. **긴 작업은 체이닝(Prompt Chaining)** — 한 프롬프트에 다 욱여넣지 말고 단계 분할

## 흔한 안티패턴

| 안티패턴 | 문제 |
|----------|------|
| "좋게 해줘", "잘 답해줘" | 기준이 없어 모델이 추측 |
| 부정문 위주 (`~하지 마`) | 모델이 부정 지시를 자주 무시. 긍정문으로 *"~을 해라"* 로 변환 |
| 한 프롬프트에 5가지 요구 | 단계 분할(체이닝)이 더 정확 |
| 예시 없이 형식 강제 | 형식이 까다로울수록 1~2개 예시 필요 |
| 너무 긴 지시문 | 모델이 핵심을 놓침. XML 태그로 구조화 |
| 매번 system 프롬프트 변경 | 캐시 히트율 저하 → 비용↑ |

## "프롬프트 엔지니어" 직무 논쟁

| 시기 | 관점 |
|------|------|
| 2023 | "AI 시대의 핵심 직무" 평가. 별도 채용 활성화 |
| 2024~2025 | 모델 발전으로 단순 프롬프트는 LLM이 알아서 잘 받음 → 직무 의문 제기 |
| 2026 (현재) | 독립 직무보다 **모든 엔지니어의 기본 스킬 + 컨텍스트/하네스 엔지니어링의 부분집합** 으로 통합 |

> 프롬프트 엔지니어링 *기술* 은 여전히 중요하지만, *별도 직무* 로서의 입지는 흐려지는 추세. 모델·도구·인프라까지 함께 다루는 컨텍스트/하네스 엔지니어링이 더 가치 있게 평가됨.
