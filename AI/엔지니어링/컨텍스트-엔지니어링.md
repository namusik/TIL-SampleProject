# 컨텍스트 엔지니어링 (Context Engineering)
> 최종 업데이트: 2026-05-02 | 기준 정보: Claude 4.x, GPT-4o, 에이전트 워크플로우 시점

## 개념
컨텍스트 엔지니어링은 **LLM의 컨텍스트 윈도우에 무엇을 / 얼마나 / 어떤 순서로 넣을지 설계하는 기술**이다. 모델에 들어가는 모든 토큰(시스템 프롬프트, 도구 정의, 이전 대화, 검색 결과, 도구 출력 등)을 하나의 "예산" 으로 보고 최적 분배하는 것이 핵심.

쉽게 비유하면 **책상 정리**와 같다. 책상(컨텍스트 윈도우)이 아무리 넓어도 잡동사니로 가득 차면 작업 효율이 떨어진다. 진짜 필요한 자료만 골라 적절한 위치에 두는 게 컨텍스트 엔지니어링이다.

## 배경/역사
- **2022~2023년**: ChatGPT 등장 후 "프롬프트 엔지니어링"이 유행. 단일 프롬프트 작성 기법이 중심
- **2023년 말**: RAG(Retrieval-Augmented Generation)가 보편화되며 "외부 지식을 컨텍스트에 어떻게 끼워넣을지" 고민 시작
- **2024년**: 에이전트(AutoGPT, LangChain Agent 등)가 등장하면서 멀티턴/도구 호출 루프에서 컨텍스트가 빠르게 부풀어 오르는 문제 부각
- **2024년 중반**: Anthropic, OpenAI가 **prompt caching** 출시 → 컨텍스트 재사용이 비용 측면에서도 중요해짐
- **2024년 말 ~ 2025년**: Anthropic 엔지니어링 블로그가 **"Context Engineering"** 용어를 공식화. Claude Code의 compaction·sub-agent 기법이 사례로 널리 알려짐
- **2025년~**: 1M 컨텍스트 윈도우(Gemini, Claude 1M 옵션) 등장 후, 오히려 "긴 컨텍스트일수록 성능이 떨어지는 현상(context rot)"이 보고되며 더욱 중요해짐

## 프롬프트 엔지니어링과의 차이

| 구분 | 프롬프트 엔지니어링 | 컨텍스트 엔지니어링 |
|------|---------------------|---------------------|
| 단위 | 한 번의 요청(프롬프트) | 워크플로우/에이전트 루프 전체 |
| 관심사 | "어떻게 잘 물어볼까?" | "무엇을 / 얼마나 컨텍스트에 둘까?" |
| 대상 | 텍스트 표현, 예시(Few-shot) | 토큰 예산 분배, 컨텍스트 구조 |
| 비용 관점 | 보통 무관 | 핵심 (토큰=비용) |
| 사용 시점 | 1회성 작업 | 장기 실행 에이전트, 챗봇, RAG |

## 컨텍스트 윈도우 구성 요소
한 요청에서 컨텍스트 윈도우를 차지하는 항목들 (위 → 아래 순서로 모델에 전달).

| # | 구성 요소 | 특성 |
|---|-----------|------|
| 1 | 시스템 프롬프트 (역할, 규칙) | 거의 고정 → 캐시 대상 |
| 2 | 도구(Tool) 정의 / 함수 시그니처 | 거의 고정 → 캐시 대상 |
| 3 | 외부 지식 (RAG 검색 결과) | 동적 |
| 4 | 이전 대화 / 에이전트 히스토리 | 누적되며 폭증 |
| 5 | 도구 호출 결과 (파일, 출력 등) | 가장 부피 큰 항목 |
| 6 | 현재 사용자 입력 | 매 요청마다 변경 |

→ 모델은 위 컨텍스트를 받아 **모델 응답(출력 토큰)** 을 생성한다.

## Anthropic 5대 원칙 (요약)
Anthropic 엔지니어링 블로그 "Effective Context Engineering"의 핵심 원칙.

1. **Just enough context** — 많을수록 좋은 게 아니라 작업에 충분한 만큼만
2. **Structure over volume** — 양보다 구조 (잘 정리된 적은 정보 > 흩어진 많은 정보)
3. **Caching first** — 캐시 가능한 부분을 앞쪽에 고정 배치
4. **Sub-agent for isolation** — 큰 탐색 작업은 격리해서 위임
5. **Compaction over deletion** — 단순 삭제 대신 요약 → 정보 손실 최소화

이 원칙들을 구현하는 구체적 기법들이 아래에 이어진다.

## 핵심 기법

### 1. 컨텍스트 선택 (Selection / Retrieval)
필요한 정보만 골라서 주입.

- **RAG**: 질문과 관련된 문서 chunk만 임베딩 유사도로 검색해서 주입
- **선별적 도구 호출 결과**: 전체 출력 대신 필요한 부분만 추출
- **다중 문서 인덱스 부여**: `<document index="1">...</document>` 형태로 참조 가능하게

### 2. 컨텍스트 구조화 (Structure & Ordering)
Anthropic은 **XML 태그로 섹션을 구분**하는 것을 강력히 권장한다. Claude는 학습 시 XML로 분리된 데이터를 많이 봤기 때문에 인식률이 높음.

배치 원칙:
- LLM은 **시작과 끝**에 있는 정보를 더 잘 기억함 (Lost-in-the-middle 현상)
- 긴 문서/배경 자료는 **컨텍스트 위쪽**(system 또는 user 시작 부분)에
- 실제 질문은 **맨 끝**에
- 정적(system, tools) → 동적(대화) → 현재 질문 순서가 일반적
- Citations 요구 (`"답변 시 원문을 인용해라"`) → hallucination 감소

```xml
<role>너는 백엔드 엔지니어를 돕는 코드 리뷰어다.</role>

<rules>
- 보안 이슈를 우선 지적한다
- 한국어로 응답한다
</rules>

<documents>
<document index="1">...첨부 코드...</document>
</documents>

<task>위 코드를 리뷰하고 보안 이슈를 우선 지적해라.</task>
```

### 3. 프롬프트 캐싱 (Prompt Caching)
- 자주 반복되는 prefix(시스템 프롬프트, 도구 정의 등)를 캐시
- 기본 TTL **5분**, 옵션으로 **1시간** (Extended cache)
- 모델별 **최소 토큰 임계치** 존재 (작은 prefix는 캐시 안 됨)
- 캐시 히트 시 입력 토큰 비용 **약 1/10** 수준
- **순서가 핵심**: 정적 → 동적 순으로 배치해야 캐시 히트율 극대화

```python
# Claude SDK 예시
messages.create(
    model="claude-opus-4-7",
    system=[
        {"type": "text", "text": LONG_SYSTEM_PROMPT,
         "cache_control": {"type": "ephemeral"}},
    ],
    tools=[...],  # 도구 정의도 캐시됨
    messages=[...]
)
```

### 4. 도구(Tool) 설계 + 출력 제어
좋은 도구 정의는 그 자체로 컨텍스트 엔지니어링이다.

설계 원칙:
- **이름**: 동사로 시작, snake_case (`read_file`, `search_users`)
- **description**: 모델이 *언제 / 왜* 호출할지 판단할 수 있을 만큼 충분히 작성
- **인자 최소화**: 필수만 required, 나머지는 optional + 기본값
- **출력 형식**: 모델이 다음 단계에서 쉽게 파싱할 수 있는 구조 (JSON, 마크다운)
- **단일 책임**: 하나의 tool은 하나의 일만 (`get_user`와 `update_user` 분리)
- **에러 메시지**: 모델이 스스로 복구할 수 있게 구체적으로 (`"id field required"` > `"invalid input"`)

출력 제어:
- 파일 읽기 결과를 **잘라서 반환** (max line, head/tail 등)
- 긴 출력은 **요약 후 저장** 하고 참조만 컨텍스트에 유지
- 검색 결과 상위 N개만 노출

### 5. 서브 에이전트 (Multi-agent Orchestration)
Anthropic이 Claude Code 등에서 사용하는 패턴. 큰 작업을 격리된 컨텍스트로 위임.

- **Lead Agent ↔ Sub-agent 분리**: 메인은 오케스트레이션, 서브는 격리된 작업 수행
- **명확한 인풋/아웃풋 정의**: 서브에 작업·결과 형식·종료 조건을 명확히 전달
- **요약만 반환**: raw 결과(파일 전체, 검색 결과 100개 등) 대신 요약만 메인 컨텍스트로
- **병렬 실행 가능**: 독립적인 탐색 작업은 동시에 여러 서브 실행
- 적용 예: Claude Code의 `Agent` tool, OpenAI Assistants의 sub-thread

### 6. 컨텍스트 압축 (Compaction)
오래된 대화·도구 결과를 **요약문으로 대체** → 정보 유지하면서 토큰 절감.

- 임계치(예: 컨텍스트 95%) 도달 시 자동 압축 (Claude Code의 auto-compact)
- 단순 truncation(앞부분 잘라내기)보다 정보 손실 적음

남기는 것 / 버리는 것:
| 보존 | 폐기 |
|------|------|
| 사용자의 핵심 지시·결정 사항 | 중간 도구 호출의 raw 결과 |
| 미해결 이슈 / TODO | 이미 완료된 단계의 상세 |
| 가장 최근 도구 호출 결과 | 오래된 탐색 결과 |
| 활성 파일·코드 상태 | 한 번 보고 지나간 파일 내용 |

## Context Rot (컨텍스트 부패)
컨텍스트가 길어질수록 성능이 떨어지는 현상.

원인:
- **노이즈 증가**: 관련 없는 정보가 쌓이며 모델 주의력 분산
- **Lost-in-the-middle**: 중간에 묻힌 정보 인식률 저하
- **토큰 누적 비용**: 매 요청마다 같은 토큰을 다시 보내야 함

대응:
- 주기적인 컨텍스트 압축
- 불필요한 도구 호출 결과 제거
- 1M 토큰을 다 채우기보다는 **꼭 필요한 50~100K만 유지**

## 토큰 예산 배분 예시

| 용도 | 권장 비율 | 비고 |
|------|-----------|------|
| 시스템 프롬프트 + 도구 정의 | 5~15% | 캐시 대상 |
| 외부 지식 (RAG) | 20~40% | 동적, 정확도 중요 |
| 대화 히스토리 | 30~50% | compaction으로 관리 |
| 현재 입력 | 5~10% | |
| 출력 여유분 | 10~20% | `max_tokens` 고려 |
