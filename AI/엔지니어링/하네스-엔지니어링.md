# 하네스 엔지니어링 (Harness Engineering)

> 최종 업데이트: 2026-05-02 | Mitchell Hashimoto가 명명(2026-02) 이후 업계 표준 용어화 시점

## 개념

하네스 엔지니어링은 **LLM이라는 "엔진"을 실용적인 에이전트로 만들기 위한 주변 인프라(도구·메모리·권한·훅·평가 등)를 설계하고 운영하는 기술 영역**이다.

> 비유: "하네스(harness)"는 원래 **말 마구·낙하산 안전벨트**를 뜻한다. LLM이라는 강력한 엔진이 폭주하지 않고 유용한 일을 하도록 **고삐를 채우고 동력을 전달하는 골격**을 만드는 일.

핵심 명제: **"같은 모델이라도 하네스 품질이 결과를 좌우한다."** Claude·GPT·Gemini 같은 모델은 점점 비슷해지고 있고, 차별화는 그 위에 무엇을 어떻게 얹느냐로 이동했다.

## 배경/역사

- **2020~2022** GPT-3·ChatGPT 등장. 단발성 프롬프트 ↔ 응답이 주류
- **2023** LangChain 같은 "프롬프트 체이닝" 프레임워크 등장. 도구 호출(Function Calling) 표준화 시작
- **2024** **Claude Code, Cursor, Devin 등 AI 에이전트** 본격 등장. 단순 챗봇이 아니라 "여러 단계 도구를 자율적으로 사용하는 시스템"으로 진화
- **2024 후반** Anthropic이 **MCP (Model Context Protocol)** 공개 — 도구·리소스 연결의 표준 프로토콜
- **2025 후반** 비슷한 실천이 곳곳에서 단편적으로 일어나기 시작. 정식 명칭은 없었음
- **2026-02** **Mitchell Hashimoto (HashiCorp 공동창업자)** 가 블로그 포스트 ["My AI Adoption Journey"](https://mitchellh.com/writing/my-ai-adoption-journey)에서 **"Engineer the Harness"** 라는 표현을 사용. 이 글이 용어의 공식 출발점
- **2026-02 직후** OpenAI가 "Harness engineering: leveraging Codex in an agent-first world" 발표. Ethan Mollick이 자신의 프레임워크를 **Models / Apps / Harnesses** 3축으로 재편. Martin Fowler가 분석 글 게재
- **2026-03~** 단 몇 주 만에 업계 표준 용어로 자리잡음. SDD 같은 방법론과 결합되어 사내 워크플로우로 정착하는 단계

> Mitchell의 핵심 정의: **"에이전트가 실수를 한 번 했으면, 같은 실수를 다시는 하지 않도록 시스템을 짜는 일"**. 단순히 "AI를 위한 인프라"라는 추상이 아니라, *실패 → 보강 → 영구 해결* 의 피드백 루프 자체를 가리킨다.

## Mitchell Hashimoto의 정의와 접근법

용어를 명명한 Mitchell Hashimoto는 하네스 엔지니어링을 **"에이전트가 실수했을 때, 같은 실수를 다시는 하지 않도록 해결책을 엔지니어링하는 것"** 으로 정의한다. 핵심 목적은 **에이전트가 한 번에 올바른 결과를 내거나, 최소한의 수정만 필요하게 만드는 것**.

### 두 가지 구현 방식

#### 1. 암시적 프롬프팅 (Implicit Prompting)
간단한 이슈는 **`AGENTS.md` / `CLAUDE.md` 같은 문서를 갱신**하여 해결.

> "그 파일의 모든 줄은 에이전트의 나쁜 행동에 기반하며, 그것들 거의 전부를 해결해줬다." — Mitchell Hashimoto

→ 문제 패턴을 발견할 때마다 해당 파일에 한 줄씩 추가. 시간이 지나면 그 자체가 "이 코드베이스에 맞는 에이전트 가이드라인"이 된다.

#### 2. 프로그래밍된 도구 (Programmed Tools)
복잡한 검증은 **자동화 스크립트**로 해결.
- 스크린샷 비교 스크립트
- 필터링된 테스트 실행 도구
- 스키마/계약 검증
- 린터/포매터 자동 실행

→ 도구를 만들고, 그 존재를 문서에 명시해서 에이전트가 활용하게 한다.

### 핵심 인사이트

각 에이전트 실패를 **"임시 수정"** 이 아니라 **"시스템 보강 기회"** 로 본다.

```
[기존 방식]                     [하네스 엔지니어링]
에이전트 실수 → 사람이 수정      에이전트 실수 → 사람이 수정 + 가드레일 추가
       ↓                              ↓
다음에 또 같은 실수             다음엔 그 실수를 못 하게 됨
       ↓                              ↓
사람이 또 수정 (반복)           시간이 지날수록 효율 ↑↑ (복리 효과)
```

이 피드백 루프가 시간이 지날수록 **복리로 효율이 누적**되는 것이 본질.

## LLM과 하네스의 관계

```
        ┌──────────────────────┐
        │     LLM (모델)        │   ← 지능 (Claude, GPT, Gemini)
        └──────────┬───────────┘
                   │ 입력/출력
        ┌──────────┴───────────┐
        │     하네스           │   ← "엔진"을 다루는 골격
        │                      │
        │  • 도구 (Tools)       │
        │  • 컨텍스트 / 메모리  │
        │  • 권한 / 샌드박스    │
        │  • 훅 (Hooks)         │
        │  • 설정 / 페르소나    │
        │  • 평가 / 재시도      │
        └──────────┬───────────┘
                   │
        ┌──────────┴───────────┐
        │   사용자 / 환경       │
        └──────────────────────┘
```

LLM은 "텍스트를 잘 생성하는 엔진"일 뿐. 실제 일(파일 편집, 명령 실행, API 호출, 다단계 작업)은 모두 하네스가 담당한다.

## 하네스 구성 요소

| 구성 요소 | 역할 | 예 |
|---|---|---|
| **도구 (Tools)** | LLM이 호출 가능한 함수 | Bash, Read, Edit, WebFetch, MCP 도구 |
| **컨텍스트 관리** | 장기 메모리, 자동 압축 | Claude Code의 `CLAUDE.md`, MEMORY 파일 |
| **권한 / 샌드박스** | 위험 명령 차단·승인 | `settings.json`의 allow/deny 룰 |
| **훅 (Hooks)** | 도구 호출 전후 자동 실행 | PreToolUse, PostToolUse, Stop |
| **페르소나 / 시스템 프롬프트** | 에이전트 성격·규칙 정의 | "한국어로 대답", "TDD로 작업" |
| **평가 루프** | 출력 검증·재시도 | 린터/테스트 자동 실행 후 피드백 |
| **다중 에이전트** | 역할 분담된 서브 에이전트 | Investigator/Auditor/Verifier |
| **로깅 / 관측** | 호출·결과 기록, 비용 추적 | Langfuse, Helicone, 자체 로그 |

## 같은 모델, 다른 하네스 — 결과 차이

| 사용 방식 | 하네스 수준 | 결과 |
|---|---|---|
| 웹 챗봇에 질문 | 거의 없음 | 단발성 답변 |
| Cursor에서 코드 편집 | 중간 (도구 + 컨텍스트) | 부분적 자동화 |
| Claude Code + CLAUDE.md + 훅 + 메모리 | 고도화 | 다단계 자율 작업 |
| 위 + 팀에이전트 + 검증 루프 | 매우 고도화 | 사내 워크플로우 자동화 |

같은 Claude 모델이라도 **하네스를 어떻게 짜느냐에 따라 "장난감"이 되기도, "팀원"이 되기도** 한다.

## 대표 사례

### Claude Code (Anthropic)
가장 잘 알려진 코딩 에이전트 하네스. **CLI + IDE 확장 + Hooks + Skills + Memory + MCP** 통합. 사내 도구로 빠르게 채택되는 중.

### Cursor / Windsurf
IDE 형태. 코드 편집·검색·실행 도구가 빌트인.

### Devin (Cognition)
"AI 소프트웨어 엔지니어"를 표방. 브라우저·에디터·셸을 통합한 가상 컴퓨터 하네스.

### LangGraph / CrewAI / AutoGen
프레임워크. **다중 에이전트 그래프**를 코드로 짤 수 있게 해주는 라이브러리.

### MCP (Model Context Protocol)
하네스의 **도구 연결 표준**. Anthropic 발표·OpenAI 채택. "USB 같은 표준 포트"를 지향.

## 단순 SDK 호출 vs 하네스 엔지니어링

| 측면 | SDK 호출 | 하네스 엔지니어링 |
|---|---|---|
| 단위 | 한 번의 API 요청 | 다단계 워크플로우 |
| 도구 | 없음 또는 단일 함수 | 다양한 도구 + 권한 |
| 메모리 | 없음 (stateless) | 세션·장기 메모리 |
| 검증 | 사람이 결과 확인 | 자동 평가·재시도 |
| 실패 처리 | 재호출 | 훅·롤백·격리 |
| 비용 | 토큰 단위 | 워크플로우 단위 |
| 관심사 | 프롬프트 품질 | 시스템 설계 품질 |

## 백엔드 개발자가 알아야 할 핵심

> 하네스 엔지니어링은 본질적으로 **"AI를 위한 백엔드 설계"**다. 익숙한 개념이 그대로 매핑된다:

| 일반 백엔드 | 하네스 |
|---|---|
| API 게이트웨이 | 권한/샌드박스 |
| 서비스 레지스트리 | 도구 카탈로그 (MCP) |
| 미들웨어 | 훅 (Hooks) |
| 캐시 / DB | 컨텍스트 / 메모리 |
| 모니터링 | LLM 관측 (Langfuse 등) |
| 회로 차단기 | 안전 차단 룰, 토큰 한도 |
| Saga / 워크플로우 | 다중 에이전트 그래프 |

즉, **익숙한 분산 시스템 패턴이 LLM 시대에 새 옷을 입은 것**. 백엔드 엔지니어 경력이 직접 도움 된다.

## 만들 때의 원칙

1. **점진적 구축** — 안 쓰는 도구·메모리·스킬을 미리 만들지 말 것. 필요할 때 만든다.
2. **결정론적 검증을 끼워라** — 린터, 테스트, 스키마 검증 같은 결정론적 도구로 LLM 출력 검사.
3. **권한은 화이트리스트** — Read-only 자동 허용, CUD는 사람 승인. 재앙 차단.
4. **컨텍스트는 잘게 쪼개라** — 큰 CLAUDE.md 하나보다 서비스별 분리가 효과적.
5. **메모리는 인덱스 + 본문** — 인덱스(MEMORY.md)는 짧게, 본문은 별도 파일로 — 컨텍스트 비용 절감.
6. **사용자 워크플로우 관찰 후 자동화** — 반복이 보일 때 스킬화. 추측으로 만들면 노이즈.
7. **평가 가능하게 설계** — 모든 단계에 "이게 잘 됐다"의 자동 판정 기준이 있어야 한다.

## 흔한 함정

| 함정 | 설명 |
|---|---|
| 권한 와일드카드 남용 | `Bash(*)` 같은 룰 → 실수로 `rm -rf` 자동 실행 |
| 메모리 모순 | "main 브랜치 기준"과 "develop 기준"이 동시에 있는 식. AI 혼란 |
| 도구 과다 | 50개 도구를 다 등록 → LLM이 도구 선택을 못함. 핵심 5~10개만 |
| 평가 누락 | 결과를 사람만 보면 자동화 의미 없음. 자동 검증 필수 |
| 컨텍스트 폭발 | 모든 정보를 매 요청에 넣음 → 비용·지연 폭증. 캐시·압축 활용 |
| 에이전트 과잉 위임 | "알아서 잘해줘" → 모호한 결과. SDD처럼 명세 우선 |

## 인접 개념

| 용어 | 관계 |
|---|---|
| **Prompt Engineering** | 단발 프롬프트 최적화. 하네스 엔지니어링의 부분집합 |
| **Agent Engineering** | 자율 에이전트 설계. 하네스 엔지니어링과 거의 동의어로 쓰임 |
| **MCP** | 하네스의 도구 연결 표준 프로토콜 |
| **SDD (Spec-Driven Dev)** | 하네스 안에서 도는 워크플로우 방법론 |
| **AI Ops** | 운영 측면 (모니터링·비용 관리). 하네스의 운영 영역 |

## 한 줄 요약

> **하네스 엔지니어링 = "AI를 위한 백엔드 설계".** 같은 LLM 모델이라도 도구·메모리·권한·훅·평가를 어떻게 짜느냐가 결과를 좌우하고, 백엔드 엔지니어의 경험(API 게이트웨이·미들웨어·관측)이 그대로 전이된다. Claude Code·Cursor·Devin이 대표 사례.
