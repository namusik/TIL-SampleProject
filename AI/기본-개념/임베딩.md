# Embedding (임베딩)

> 최종 업데이트: 2026-03-27

## 개념

텍스트, 이미지, 오디오 등의 데이터를 **의미를 담은 숫자 배열(벡터)** 로 변환하는 것.

- 사람은 "강아지"와 "puppy"가 비슷하다는 걸 알지만, 컴퓨터는 문자열로는 판단 불가
- 임베딩은 이런 **의미적 유사성을 숫자로 표현**하여 컴퓨터가 계산할 수 있게 함
- 지도 위 좌표에 비유하면: "서울"과 "인천"은 좌표가 가깝고, "서울"과 "뉴욕"은 멀듯이, 의미가 비슷한 데이터는 벡터 공간에서 가까운 위치에 놓임

```
"오늘 날씨가 좋다"  → [0.12, -0.34, 0.56, 0.78, ...]
"날씨가 화창하다"   → [0.11, -0.33, 0.55, 0.80, ...]   ← 의미 비슷 → 벡터 가까움
"주식이 올랐다"     → [0.91, 0.22, -0.45, 0.10, ...]   ← 의미 다름 → 벡터 멀어짐
```

## 벡터 변환의 기준

사람이 규칙을 정한 것이 아니라, 임베딩 모델이 **대량의 데이터로 학습하면서 스스로 기준을 만든 것**.

- 시험공부에 비유하면: 문제를 수억 개 풀면서 **"이런 패턴은 이렇게 분류하면 되는구나"를 스스로 체득**하는 것. 정답 공식이 있는 게 아니라 경험으로 감을 잡은 것

### 학습 원리

모델이 학습할 때 핵심 규칙: **의미가 비슷한 것은 가깝게, 다른 것은 멀게** 배치되도록 벡터 변환 방식을 조정

```
학습 데이터: 수억 개의 문장 쌍

"강아지가 뛰어논다" ↔ "puppy is running"     → 비슷하다고 학습
"강아지가 뛰어논다" ↔ "주식 시장이 폭락했다"   → 다르다고 학습

→ 수억 번 반복하면서 벡터 변환 규칙(가중치)을 자동으로 학습
```

### 벡터의 각 숫자는 뭘 의미하는가

"1번째 값 = 감정, 2번째 값 = 시제" 같은 명확한 대응은 **없다**.

- 모델이 학습 과정에서 자체적으로 만든 추상적 특징(feature)
- 개별 숫자 하나는 의미가 없고, **벡터 전체의 조합**으로 의미를 표현
- 그래서 임베딩 모델마다 같은 텍스트를 넣어도 벡터가 다름 — 모델마다 학습 데이터와 구조가 달라 각자 다른 변환 기준을 가짐

## 임베딩 모델

텍스트를 벡터로 변환해주는 AI 모델. 번역기가 한국어를 영어로 바꾸듯, 임베딩 모델은 텍스트를 숫자 배열로 바꿔줌.

| 모델 | 제공 | 차원 수 | 특징 |
|------|------|---------|------|
| text-embedding-3-small | OpenAI | 1536 | 가성비 좋음, 범용 |
| text-embedding-3-large | OpenAI | 3072 | 높은 정확도 |
| Cohere Embed v3 | Cohere | 1024 | 다국어 지원 우수 |
| BGE-M3 | BAAI | 1024 | 오픈소스, 다국어 |
| all-MiniLM-L6-v2 | Sentence Transformers | 384 | 오픈소스, 경량 |

- **차원 수** — 벡터의 길이. 높을수록 더 세밀한 의미 표현이 가능하지만 저장/연산 비용 증가
- 용도에 따라 선택: 빠른 프로토타이핑이면 small/경량 모델, 정확도가 중요하면 large 모델

## 임베딩 파이프라인

실무에서 임베딩을 활용할 때의 전체 흐름.

```
[저장]
원본 문서 → 청크 분할 → 임베딩 모델 → 벡터 → 벡터DB에 저장

[검색]
사용자 질문 → 임베딩 모델 → 질문 벡터 → 벡터DB에서 유사도 검색 → 결과 반환
```

### 청크 분할 (Chunking)

문서를 적절한 크기의 **조각(chunk)으로 나누는 단계**. 임베딩 전에 반드시 필요.

- 책 한 권을 통째로 요약하면 핵심이 흐려지지만, **챕터별로 요약하면 각각의 핵심을 정확히 찾을 수 있는 것**과 같음
- 임베딩 모델에 입력 길이 제한이 있고, 너무 큰 텍스트를 하나의 벡터로 만들면 의미가 뭉뚱그려져 검색 정확도 저하

```
[원본 - 사내 휴가 정책 문서]

→ 청크 분할:
  청크1: "연차는 입사일 기준으로 매년 15일이 부여되며..."
  청크2: "병가는 유급으로 연 60일까지 사용 가능하며..."
  청크3: "경조사 휴가는 결혼 5일, 출산 10일..."

→ 각각 임베딩하여 벡터DB에 저장

→ "출산 휴가 며칠이야?" 질문 시 → 청크3이 가장 유사 → 청크3을 LLM에 전달
```

#### 분할 전략

| 방식 | 설명 | 적합한 경우 |
|------|------|------------|
| 고정 크기 | 일정 토큰/글자 수로 자름 (예: 500토큰씩) | 빠른 구현, 균일한 청크 필요 시 |
| 오버랩 | 청크 간 일부를 겹치게 분할 | 문맥 유실 방지가 중요할 때 |
| 의미 단위 | 문단, 섹션, 마크다운 헤더 등 경계로 분할 | 구조화된 문서 |

- 청크가 **너무 크면** 검색 정확도 저하, **너무 작으면** 문맥 부족
- 실무에서는 보통 **300~1000토큰 + 오버랩**으로 시작해서 조정

## 유사도 측정

두 벡터가 얼마나 가까운지 계산하는 방법. 쇼핑몰에서 "이 상품과 취향이 얼마나 비슷한가"를 수치로 매기는 것과 같음.

| 방식 | 설명 | 주 사용처 |
|------|------|----------|
| Cosine Similarity | 벡터 간 각도 기반 | 텍스트 검색 (가장 보편적) |
| Euclidean Distance | 벡터 간 직선 거리 | 값의 크기가 중요할 때 |
| Dot Product | 벡터 내적 | 방향 + 크기 모두 반영 |

## 활용 사례

| 사례 | 설명 |
|------|------|
| RAG | LLM이 답변 전 관련 문서를 임베딩 검색으로 찾아 정확도 향상 |
| 시맨틱 검색 | "저렴한 숙소" → "가성비 좋은 호텔"도 매칭되는 의미 기반 검색 |
| 추천 시스템 | 유저 행동·상품을 벡터화하여 유사 콘텐츠 추천 |
| 분류/클러스터링 | 벡터 간 거리로 문서를 자동 분류·그룹화 |
| 이상 탐지 | 정상 패턴 벡터와 거리가 먼 데이터를 이상치로 판별 |
