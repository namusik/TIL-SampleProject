# 컨텍스트 (Context)
> 최종 업데이트: 2026-05-02 | 기준 정보: GPT-4o, Claude 4.x, Gemini 1.5 시점

## 개념
컨텍스트는 **LLM이 한 번의 호출에서 볼 수 있는 입력 텍스트 전체**다. 모델은 매 요청마다 컨텍스트만 보고 답변을 생성한다.

쉽게 비유하면 **매번 기억상실증에 걸린 천재 박사**와 같다. 박사는 일 자체는 잘 하지만, 매번 시킬 때마다 "어제 우리가 무슨 얘기 했고, 지금 상황은 이렇고..." 라고 노트를 다시 건네야 한다. 그 노트 한 장이 바로 컨텍스트다.

## 배경/역사
- **2017년**: Transformer 논문 ("Attention is All You Need") — Self-attention 구조의 입력 길이 제한이 컨텍스트 윈도우 개념의 시작
- **2020년**: GPT-3 등장. 컨텍스트 윈도우 2K → 4K로 확장되며 "프롬프트에 얼마나 넣을 수 있나"가 실무 이슈로 부상
- **2023년**: GPT-4 (8K~32K), Claude 2 (100K) 출시. 긴 문서 첨부가 가능해지며 컨텍스트 활용법이 다양해짐
- **2024년**: Claude 3 (200K), Gemini 1.5 (1M~2M) 등장. "긴 컨텍스트 시대" 도래
- **2024년 말 ~ 2025년**: Anthropic이 **Context Engineering** 개념을 공식화. 양보다 구조·관리가 중요하다는 인식 확산
- **2025년**: 1M 컨텍스트가 보편화되었으나 **Context Rot** (긴 컨텍스트일수록 성능 저하) 문제도 함께 알려짐

## LLM은 기억하지 않는다 (Stateless)
LLM의 가장 핵심적인 특성:

- 모델은 **stateless** — 이전 호출을 기억하지 않는다
- 채팅 앱이 "기억"하는 것처럼 보이는 이유는 **앱이 매 요청마다 이전 대화를 컨텍스트에 다시 넣어서 보내기 때문**
- 모델 입장에서는 매 요청이 처음 보는 입력. 그 한 번의 입력 안에 과거가 적혀있을 뿐

```
[1번째 요청]
컨텍스트: "안녕"
응답: "안녕하세요!"

[2번째 요청]  ← 이전 대화를 다시 포함해서 보내야 함
컨텍스트: "안녕\n안녕하세요!\n오늘 날씨 어때?"
응답: "오늘은..."

[3번째 요청]  ← 또 다시 누적
컨텍스트: "안녕\n안녕하세요!\n오늘 날씨 어때?\n오늘은...\n고마워"
응답: "별 말씀을요!"
```

→ 그래서 **대화가 길어질수록 매 요청의 컨텍스트가 폭발적으로 커진다**. 이것이 "토큰이 녹는다"의 본질.

## 컨텍스트의 구성 요소
한 요청의 입력 컨텍스트는 아래 항목들의 합으로 구성된다 (위 → 아래 순서로 모델에 전달).

| # | 구성 요소 | 설명 | 특성 |
|---|-----------|------|------|
| 1 | 시스템 프롬프트 | 모델 역할 / 규칙 정의 | 거의 고정 (캐시 대상) |
| 2 | 도구(Tool) 정의 | 모델이 호출 가능한 함수 시그니처 | 거의 고정 (캐시 대상) |
| 3 | 첨부 파일 / 외부 지식 (RAG) | PDF, 검색 결과 등 | 동적 |
| 4 | 이전 대화 / 에이전트 히스토리 | 멀티턴 누적 | 누적되며 폭증 |
| 5 | 도구 호출 결과 | 파일 읽기, API 응답 등 | 가장 부피 큰 항목 |
| 6 | 현재 사용자 입력 | 이번 질문 | 매 요청마다 변경 |

→ 모델은 위 컨텍스트를 받아 **모델 응답(출력 토큰)** 을 생성한다.

## 컨텍스트 윈도우 (Context Window)
모델이 한 번에 받을 수 있는 컨텍스트의 **최대 토큰 수** (입력 + 출력 합산).

| 모델 | 컨텍스트 윈도우 |
|------|-----------------|
| GPT-3.5 Turbo | 16K |
| GPT-4 Turbo / GPT-4o | 128K |
| Claude 3.5 Sonnet | 200K |
| Claude 4.x | 200K (1M 옵션) |
| Gemini 1.5 Pro | 1M ~ 2M |

> 200K 토큰 ≈ 한국어 책 약 300~400페이지 분량. 컨텍스트 윈도우를 초과하면 API 에러가 발생한다.

## 토큰과의 관계

| 개념 | 의미 |
|------|------|
| 토큰 | 컨텍스트를 측정하는 최소 단위 |
| 컨텍스트 | 한 요청에서 모델에 들어가는 토큰들의 묶음 |
| 컨텍스트 윈도우 | 컨텍스트 토큰 수의 상한 |

"컨텍스트가 50K 토큰이다" = 이번 요청에 50K 분량의 입력이 들어갔다는 뜻.

## 관련 용어 정리

| 용어 | 뜻 |
|------|-----|
| **컨텍스트 윈도우 (Context Window)** | 한 번에 넣을 수 있는 컨텍스트의 최대 크기 |
| **컨텍스트 엔지니어링 (Context Engineering)** | 컨텍스트 안에 무엇을 어떻게 채울지 설계하는 기술 |
| **컨텍스트 누적** | 대화·에이전트 루프에서 컨텍스트가 부풀어 오르는 현상 |
| **컨텍스트 압축 (Compaction)** | 오래된 내용을 요약해 컨텍스트 줄이기 |
| **컨텍스트 캐싱 (Prompt Caching)** | 자주 반복되는 prefix를 캐시해 비용 절감 |
| **Context Rot** | 컨텍스트가 길어질수록 성능이 저하되는 현상 |
| **Lost-in-the-middle** | 컨텍스트 중간에 위치한 정보를 모델이 잘 못 찾는 현상 |

## 흔한 오해
1. **"AI가 우리 대화를 기억해요"** → 모델이 아니라 앱이 기억해서 매번 다시 보내주는 것
2. **"컨텍스트 윈도우가 클수록 무조건 좋다"** → Context Rot 때문에 꼭 그렇지 않음. 양보다 구조와 정확도가 중요
3. **"토큰 = 컨텍스트"** → 토큰은 단위, 컨텍스트는 묶음. 다른 개념
4. **"긴 컨텍스트는 무료다"** → 매 요청마다 입력 토큰 비용이 청구됨. 캐싱이 필수
