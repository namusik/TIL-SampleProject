# 쿠버네티스 네트워킹

## DNS (CoreDNS)

### DNS란 무엇인가

- DNS(Domain Name System) 는 이름을 IP 주소로 변환해주는 시스템
- 쿠버네티스(EKS)에서는 DNS가 특히 중요
  - 파드 IP는 수시로 변경
  - 서비스는 이름으로 통신해야 안정적
  - 외부 리소스(RDS, 외부 API) 접근 시도 이름 기반

#### 쿠버네티스 내부 DNS 예시

```text
my-service.my-namespace.svc.cluster.local
```
- my-service → Service 이름
- my-namespace → 네임스페이스
- cluster.local → 클러스터 도메인

파드에서 위 이름으로 요청하면, DNS가 Service의 ClusterIP로 변환해준다.

### EKS에서 DNS는 누가 담당하는가?

- EKS에서는 기본적으로 CoreDNS 가 DNS 역할을 한다.
  - CoreDNS는 Deployment 형태로 실행
  - kube-system 네임스페이스에 존재
  - AWS EKS가 기본 설치 및 초기 설정 제공

즉, “EKS의 DNS” = CoreDNS

### DNS 레이어

### 클러스터 내부 DNS (Internal DNS)

- 담당: CoreDNS
- 역할:
  - Service → ClusterIP
  - Pod → Pod IP
- 예시
  - my-api.default.svc.cluster.local


### 클러스터 외부 DNS (External DNS)

- 담당: Route53, Cloud DNS 등
- 역할:
  - 외부 도메인 → LoadBalancer / Ingress IP
- 예시:
  - api.example.com → ALB
- external-dns의 역할 영역

### external-dns add-on이 없을 때 = 왜 수동 관리인가

- external-dns가 없는 경우 흐름은 다음과 같습니다.
1. Ingress / Service 생성
2. ALB 또는 NLB 생성됨
3. 운영자가 직접
   1. Route53에 A / CNAME 레코드 생성
   2. 변경 시 수동 수정
   3. 삭제 시 수동 정리

즉, “쿠버네티스 리소스 상태와 외부 DNS 상태가 자동으로 동기화되지 않는다”

### external-dns를 설치하면 뭐가 자동화되는가

- external-dns 는 다음 역할만 담당
- Kubernetes 리소스를 감시
  - Ingress
    - Ingress에 hostname annotation을 달면 ExternalDNS가 Route53 레코드를 자동 관리
```annotations:
    external-dns.alpha.kubernetes.io/hostname: api.example.com
```
  - Service (type=LoadBalancer)
- annotation 기반으로
  - Route53 레코드 자동 생성
  - 변경 시 자동 업데이트
  - 삭제 시 자동 제거

외부 DNS만 자동화
