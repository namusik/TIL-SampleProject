# linux 저장 공간 

## 디스크 동작 철학
### 파일과 이름의 분리

- 리눅스에서는 파일의 '이름'과 '실제 데이터(알맹이)'가 별도로 관리됨.
  - 알맹이 (Inode): 파일의 실제 데이터와 소유자, 권한 등 모든 정보가 담겨있는 본체
  - 이름표 (Dentry): 우리가 ls로 보는 파일 이름. 이 이름표는 특정 알맹이(Inode)를 가리키는 포인터 역할
- 하나의 알맹이에 여러 개의 이름표를 붙일 수도 있다. (하드 링크).

### rm 명령어의 실제 동작
- 우리가 rm -f 명령어를 사용하면, 파일의 '알맹이'를 직접 지우는 것이 아니라 '이름표'를 떼어내는 동작을 한다.
- 운영체제는 해당 파일의 알맹이(Inode)를 더 이상 가리키는 이름표가 없고, 그 알맹이를 사용 중인 프로세스도 없을 때, 비로소 "아무도 쓰지 않으니 이제 버려도 되겠다"고 판단하여 디스크 공간을 회수

### 장점이자 단점

- 장점 (안정성): 
  - 어떤 프로세스가 파일을 열심히 사용하고 있는데 다른 사용자가 그 파일을 rm으로 지워도, 실행 중인 프로세스는 아무 문제 없이 작업을 계속할 수 있습니다. 시스템이 매우 안정적으로 동작하게 됩니다.
- 단점 (사용자의 오해): 
  - 하지만 사용자는 파일을 지웠다고 생각하는데 디스크 용량은 줄어들지 않는, 지금 겪고 계신 현상이 발생. 로그 파일처럼 장시간 열려있는 파일을 올바른 방법(프로세스 재시작 또는 truncate)으로 처리하지 않으면 이런 문제가 생깁니다.

## /proc

- 리눅스 시스템의 특별한 디렉토리로, **실제 디스크에 존재하는 '진짜' 파일 시스템이 아니다.**
- 대신, 커널(Kernel)이 메모리에서 관리하는 시스템의 다양한 정보와 프로세스 상태를 파일 형태로 보여주는 **'가상' 파일 시스템(Virtual Filesystem)**
- 자동차의 계기판 예시
  - 계기판은 엔진이나 연료 탱크 그 자체가 아니지만, 현재 엔진의 RPM이나 남은 연료량 같은 실시간 정보를 보여준다.
  - 마찬가지로 /proc은 커널 그 자체가 아니지만, 현재 실행 중인 프로세스 목록, CPU 정보(cat /proc/cpuinfo), 메모리 상태(cat /proc/meminfo) 등 커널이 알고 있는 실시간 정보를 파일처럼 보여주는 통로입니다.
- 이 디렉토리의 파일들은 우리가 접근할 때마다 커널이 즉석에서 정보를 만들어 보여주며, 디스크 공간을 차지하지 않는다.


## 명령어

```sh
[ec2-user@ip-10-0-137-112 log]$ df -h
Filesystem      Size  Used Avail Use% Mounted on
devtmpfs        3.9G     0  3.9G   0% /dev
tmpfs           3.9G     0  3.9G   0% /dev/shm
tmpfs           3.9G  395M  3.5G  11% /run
tmpfs           3.9G     0  3.9G   0% /sys/fs/cgroup
/dev/nvme0n1p1  8.0G  8.0G   20K 100% /
/dev/nvme1n1    1.0T   27G  998G   3% /data
tmpfs           785M     0  785M   0% /run/user/1000
tmpfs           785M     0  785M   0% /run/user/0

sudo du -sh * | sort -hr
```


### /dev/nvme0n1p1
-	**/dev/nvme0n1p1**는 EC2 인스턴스에 연결된 첫 번째 NVMe(EBS) 볼륨입니다.
-	nvme0n1은 NVMe 프로토콜을 사용하는 첫 번째 블록 장치를 의미하고, 그 뒤의 p1은 이 디스크에서 첫 번째 파티션을 나타냅니다.
-	보통 이 파티션은 루트 파일 시스템(/)으로 마운트됩니다. 즉, 운영 체제와 중요한 시스템 파일들이 이 파티션에 저장됩니다.
-	이 파티션이 100%로 꽉 차면 시스템 성능이나 안정성에 문제가 발생할 수 있습니다.
- 로그 파일 정리
  - 주로 /var/log 디렉토리에 있는 로그들 정리
- 임시 파일 삭제
  - /tmp, /var/tmp 디렉토리의 불필요한 임시 파일 정리
  - 패키지 매니저 캐시 정리 (예: sudo yum clean all, sudo apt clean)


### /dev/nvme1n1
-	**/dev/nvme1n1**은 EC2 인스턴스에 연결된 두 번째 NVMe(EBS) 볼륨입니다.
-	nvme1n1은 두 번째 NVMe 프로토콜 블록 장치를 의미하며, 여기에는 별도의 파일 시스템이 있을 수 있습니다. 예를 들어, 데이터를 저장하는 /data와 같은 디렉토리가 이 파티션에 마운트될 수 있습니다.
-	이 볼륨은 용량이 크고 사용량이 적어 데이터를 저장하는 용도로 많이 사용됩니다.


### 용량 확인

```sh
du -h --max-depth=1 | sort -hr
```
- du: 디스크 사용량을 보여주는 명령어
- -h: 용량을 사람이 읽기 쉬운(Human-readable) 단위인 K(킬로바이트), M(메가바이트), G(기가바이트)로 보여줌
- --max-depth=1: 현재 위치를 기준으로 최대 1단계 깊이의 디렉토리까지만 용량을 보여준다. 이 옵션이 없으면 모든 하위 디렉토리의 용량을 전부 보여주어 결과가 너무 복잡
- sort: 입력된 텍스트를 정렬하는 명령어
- -h: du -h로 출력된 K, M, G 같은 단위를 숫자로 인식하여(Human-numeric sort) 크기 순서대로 올바르게 정렬
- -r: 역순(Reverse)으로 정렬하여, 용량이 가장 큰 항목이 맨 위로 오도록

```sh
ls -lhrS
```
- 하위 디렉토리는 제외하고 바로 아래에 있는 파일들의 용량만 보고 싶을 때
- ls -l: 파일을 자세한 목록 형태로 보여줍니다.
- -h: 파일 크기를 사람이 읽기 쉬운 단위(G, M, K)로 보여줍니다.
- -r: 정렬 순서를 반대로 (내림차순) 합니다.
- -S: 파일을 크기순으로 정렬합니다.

### 삭제 전 확인: 프로세스가 파일을 사용 중인지 확인

```sh
# 특정 파일을 사용 중인 프로세스가 있는지 확인
lsof [파일경로]

# 예시
lsof /var/log/app.log
```
- 결과가 출력되면 → 해당 프로세스가 파일을 열고 있는 상태이므로, rm 대신 truncate -s 0을 쓰거나 프로세스를 재시작해야 한다.
- 결과가 없으면 → 아무 프로세스도 사용 중이 아니므로 rm으로 삭제하면 즉시 용량이 반환된다.

```sh
# fuser로도 확인 가능 (파일을 사용 중인 프로세스의 PID만 간단히 확인)
fuser [파일경로]

# 자세한 정보 포함
fuser -v [파일경로]
```

### 일반적인 파일 삭제 (프로세스가 사용하지 않는 파일)

```sh
# 기본 삭제
rm [파일명]

# 묻지 않고 강제 삭제
rm -f [파일명]
```
- 리눅스에서 rm은 파일을 실제로 지우는 명령이 아니라, 파일 이름과 inode의 연결을 끊는(unlink) 작업
- 일반적인 경우
  - 파일을 어떤 프로세스도 사용 중이 아님
  - rm 파일명 실행
  - inode 참조 수 = 0
  - 즉시 디스크 공간 반환

👉 이 경우:
- df -h 기준 용량 바로 감소. 물리적으로도 공간 확보됨

### rm -f로 파일을 삭제해도 용량이 줄지 않은 케이스

- 프로세스가 파일을 사용 중인 상태에서 rm 한 경우 발생 (위 "디스크 동작 철학" 참고)
- 겉보기에는 파일이 지워졌지만, 디스크 용량은 그대로 차지하는 '좀비 파일' 상태
- 프로세스는 파일 디스크립터(FD)를 통해 Inode에 직접 연결되어 있으므로, 이름표가 사라져도 실제 데이터를 정상적으로 계속 기록한다. 오히려 로그가 계속 쌓여 용량이 더 늘어날 수도 있다.
- 해당 파일을 열고 있는 프로세스가 종료될 때 비로소 용량이 반환됨

#### 좀비 파일 확인

```sh
sudo lsof | grep deleted | sort -k 7 -n
```
- sudo lsof: 시스템에서 현재 열려있는 모든 파일의 목록을 보여주는 명령어
- grep deleted: 그 목록 중에서 (deleted) 라고 표시된, 즉 파일 이름은 삭제되었지만 프로세스가 여전히 붙잡고 있어 디스크 공간은 차지하고 있는 파일들만 골라서 보여줌
- sort -k 7 -n: 7번째 컬럼(파일 크기)을 기준으로 숫자 정렬하여, 용량을 많이 차지하는 순서로 확인 가능

#### 해결방법

**1. 프로세스 재시작이 가능한 경우 (Best)**
- 해당 프로세스를 재시작하면 기존에 붙잡고 있던 (deleted) 파일과의 연결이 끊기고 디스크 공간이 완전히 회수됨

**2. 프로세스 재시작이 불가능한 경우 (/proc 비상 통로)**
- /proc 디렉토리 안에는 실행 중인 각 프로세스의 정보가 PID를 이름으로 하는 하위 디렉토리에 저장됨
  - 예: PID가 470인 프로세스 → /proc/470/
- /proc/[PID]/fd/ 안에는 해당 프로세스가 열고 있는 모든 파일들이 파일 디스크립터(FD) 번호를 이름으로 하여 링크처럼 존재
  - 즉, 원래의 이름표는 사라졌지만 /proc/470/fd/3 이라는 커널과 직접 통하는 또 다른 이름표가 남아있는 것
- 이 링크를 통해 truncate하면, 프로세스를 건드리지 않고 파일의 내용물만 비워서 디스크 공간을 회수할 수 있다.
```sh
sudo truncate -s 0 /proc/[PID]/fd/[fd번호]
```

### 프로세스가 사용 중인 파일을 안전하게 삭제하는 방법

- 가장 권장하는 방법: **logrotate** 유틸리티 사용
  - 리눅스의 거의 모든 배포판에 포함된 로그 관리 전문 유틸리티
  - 설정 파일에 규칙(copytruncate 등)만 지정해두면, 삭제, 압축, 보관 주기 관리까지 알아서 가장 안전한 방식으로 처리해 줌
  - 직접 스크립트를 짜는 것보다 훨씬 안정적이고 강력하게 권장되는 표준 방식
- logrotate를 쓸 수 없는 상황이라면 다음 순서로 처리하는 것이 좋다.
  - 파일이 아직 존재하는 경우 → 내용만 비우기
```sh
truncate -s 0 [파일명]
```
  - 이렇게 하면 프로세스를 중단시키지 않고도 디스크 공간을 즉시 확보 가능
  - 이미 rm으로 지워버린 경우 → 위의 "/proc 비상 통로" 방법 사용


## df와 du의 용량 차이

- `df`와 `du`는 용량을 측정하는 관점이 다르기 때문에 결과가 다를 수 있다.
  - **df**: 파일시스템 레벨에서 측정. 파일시스템이 "사용 중"이라고 표시한 전체 블록을 기준으로 계산
  - **du**: 파일 레벨에서 측정. 실제로 이름이 존재하는 파일들의 크기를 합산
- 좀비 파일이 있으면 두 명령어의 결과가 크게 차이남
  - df는 프로세스가 붙잡고 있는 (deleted) 파일의 공간도 "사용 중"으로 집계
  - du는 이름이 사라진 파일은 집계하지 못함
  - 따라서 **df 결과 > du 결과** 가 되는 현상이 발생

```sh
# df와 du 비교로 좀비 파일 존재 여부를 간접적으로 확인
df -h /
du -sh /

# 두 결과의 차이가 크면 좀비 파일이 디스크를 잡아먹고 있을 가능성이 높다
```

## Inode 고갈

### Inode란
- 파일 하나당 Inode 하나가 할당됨
- 파일시스템을 생성할 때 Inode의 총 개수가 정해지며, 이후에는 늘릴 수 없다 (ext4 기준)
- 즉, 디스크 용량과 별개로 Inode에도 한도가 있다

### 증상
- 디스크 용량은 충분히 남아있는데 파일 생성이 안 되는 경우
- "No space left on device" 에러가 발생하지만 df -h로 보면 여유 공간이 있음
- 주로 작은 파일이 수십~수백만 개 쌓일 때 발생 (예: 세션 파일, 캐시 파일, 메일 큐 등)

### 확인 방법

```sh
df -i
```
- -i: Inode 사용량을 보여주는 옵션
- IUse%가 100%이면 Inode가 고갈된 상태

```sh
# 어떤 디렉토리에 파일이 많은지 확인
# 각 디렉토리의 파일/디렉토리 개수를 세어서 많은 순으로 정렬
find [경로] -xdev -type f | cut -d "/" -f 2-3 | sort | uniq -c | sort -rn | head -20
```

### 해결 방법
- 불필요한 파일을 대량 삭제하여 Inode를 회수
- 근본적으로는 작은 파일이 대량 생성되는 원인을 찾아서 해결 (캐시 정리 크론잡, 세션 만료 설정 등)


## Swap 공간

### Swap이란
- 물리 메모리(RAM)가 부족할 때, 디스크의 일부를 임시 메모리처럼 사용하는 공간
- RAM에서 당장 사용하지 않는 데이터를 디스크로 옮겨두고(swap out), 필요할 때 다시 가져오는(swap in) 방식
- 디스크는 RAM보다 훨씬 느리므로 swap 사용량이 높으면 시스템 성능이 크게 저하됨

### 확인 방법

```sh
# 메모리와 swap 사용량을 한눈에 확인
free -h
```
- total: 전체 swap 크기
- used: 현재 사용 중인 swap 크기
- free: 남은 swap 크기

```sh
# swap 파티션/파일 상세 정보
swapon --show
```

### swap 사용량이 높을 때
- 물리 메모리가 부족하다는 신호이므로, 메모리를 많이 사용하는 프로세스를 확인
```sh
# 메모리 사용량 순으로 프로세스 정렬
ps aux --sort=-%mem | head -20
```
- 근본적인 해결: 메모리 증설(인스턴스 스펙 업) 또는 메모리 누수가 있는 프로세스 수정
- swap 자체를 늘리는 것은 임시방편이지 근본 해결이 아님
