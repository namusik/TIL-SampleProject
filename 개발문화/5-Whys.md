# 5 Whys

> 최종 업데이트: 2026-04-09

## 개념

문제의 **근본 원인(Root Cause)**을 찾기 위해 "왜?"를 반복하는 분석 기법이다.

- 병원에 비유하면, 두통(표면 증상)에 진통제만 주는 게 아니라 "왜 두통이 생겼는가"를 반복 추적하여 **원인 질환**을 찾는 것
- 보통 **5회** 반복하면 근본 원인에 도달한다고 하여 5 Whys지만, 횟수는 고정이 아님. 3회로 충분할 수도, 7회가 필요할 수도 있다
- 표면적 원인에 대증 처방하면 같은 장애가 반복되고, 근본 원인을 해결해야 **재발을 방지**할 수 있다

```
문제 발생
  │
  ├── 왜? → 1차 원인 (표면)
  │     ├── 왜? → 2차 원인
  │     │     ├── 왜? → 3차 원인
  │     │     │     ├── 왜? → 4차 원인
  │     │     │     │     └── 왜? → 근본 원인 (Root Cause)
  │     │     │     │              └── 대책 수립
```

## 배경

| 항목 | 내용 |
|---|---|
| 창시자 | **사키치 토요다**(豊田 佐吉), 토요타 자동차 창업자 |
| 도입 | 1930년대 토요타 생산 방식(**TPS**, Toyota Production System) |
| 확산 | 린(Lean) 제조, 식스 시그마를 거쳐 **소프트웨어 엔지니어링**으로 확산 |
| 현재 | Google, Amazon, Netflix 등 빅테크의 **포스트모템(Post-Mortem)** 표준 도구 |

## 진행 방법

### 1. 문제를 명확하게 정의

모호한 "서버가 느려요"가 아니라 구체적으로 기술한다.

```
❌ 서버가 느려요
✅ 2026-04-08 14:00~14:30, 주문 API(/api/orders) 응답 시간이 평균 200ms → 8초로 증가
```

### 2. "왜?"를 반복

각 답변은 **사실과 데이터**에 기반해야 한다. 추측이 아닌 로그, 메트릭, 코드를 근거로 삼는다.

### 3. 근본 원인에서 대책 도출

근본 원인은 보통 **프로세스, 시스템 설계, 문화** 수준에 도달한다. 개인의 실수가 아니라 **그 실수를 허용한 구조**가 근본 원인이다.

## 장애보고서 예시

### 예시 1: API Timeout

```
장애: 주문 API 504 Gateway Timeout 발생 (2026-04-08 14:00~14:30)

1. 왜 504가 발생했는가?
   → DB 쿼리가 30초 이상 걸려 Nginx proxy_read_timeout(60s)을 초과

2. 왜 쿼리가 30초나 걸렸는가?
   → orders 테이블에서 Full Table Scan이 발생

3. 왜 Full Scan이 발생했는가?
   → 새로 추가한 필터(status + created_at) 조합에 인덱스가 없었음

4. 왜 인덱스를 걸지 않았는가?
   → 개발 시 EXPLAIN으로 실행 계획을 확인하지 않았음

5. 왜 실행 계획을 확인하지 않았는가?
   → 쿼리 변경 시 실행 계획 검토 프로세스가 없었음
   ─────────────────────────────────────────
   근본 원인: 쿼리 변경에 대한 리뷰 프로세스 부재
   대책: PR에 쿼리 변경 시 EXPLAIN 결과 첨부 의무화
```

### 예시 2: 배포 장애

```
장애: 배포 후 로그인 기능 전체 장애 (2026-04-05 10:00~10:45)

1. 왜 로그인이 안 됐는가?
   → 세션 저장소(Redis) 연결 실패

2. 왜 Redis 연결이 실패했는가?
   → 환경변수 REDIS_HOST가 빈 값이었음

3. 왜 환경변수가 비어 있었는가?
   → 신규 서버 인스턴스에 환경변수가 주입되지 않았음

4. 왜 주입되지 않았는가?
   → Terraform 코드에 해당 변수가 누락되어 있었음

5. 왜 Terraform에서 누락되었는가?
   → 인프라 변경과 앱 설정 변경이 별도 PR로 나뉘어 동기화가 안 됨
   ─────────────────────────────────────────
   근본 원인: 앱-인프라 설정 변경의 원자성 미보장
   대책: 앱 환경변수 변경 시 Terraform 변경 PR을 연결 필수화
         + 배포 전 환경변수 검증 스크립트 추가
```

## 5 Whys 작성 흐름도

```
장애 발생
     │
     ▼
┌─────────────────────────┐
│ 1. 문제 정의             │
│    (시간, 영향 범위, 증상) │
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│ 2. 타임라인 작성          │    ← 로그, 메트릭, 알림 기록 수집
│    (발생 → 감지 → 복구)   │
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│ 3. 5 Whys 수행           │    ← 사실 기반, 추측 금지
│    "왜?" 반복             │
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│ 4. 근본 원인 도출         │    ← 개인 실수 ✕, 시스템/프로세스 ✓
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│ 5. 대책 수립 (Action Item)│    ← 담당자, 기한 명시
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│ 6. 포스트모템 공유         │    ← 팀 전체에 공유, Blame-free
└─────────────────────────┘
```

## 주의사항

| 항목 | 설명 |
|---|---|
| **사람을 탓하지 않는다** | "왜 실수했어?"가 아니라 "왜 그 실수가 가능한 구조였는가?"를 묻는다. Blame-free 문화가 전제 |
| **추측이 아닌 데이터** | 각 "왜?"의 답은 로그, 메트릭, 코드 등 **증거**에 기반. "아마 ~일 것이다"는 위험 |
| **한 줄기로 가지 않는다** | 원인이 여러 갈래일 수 있음. 필요하면 **분기(Fishbone)**하여 여러 경로 추적 |
| **5회에 집착하지 않는다** | 3회로 충분하면 멈추고, 7회가 필요하면 계속. 핵심은 **근본 원인 도달** |
| **대책은 구체적으로** | "더 주의한다"는 대책이 아님. 자동화, 프로세스 추가, 모니터링 설정 등 **검증 가능한** 대책 |

## 보완 기법

5 Whys만으로 부족할 때 함께 쓰는 도구들.

| 기법 | 설명 | 활용 |
|---|---|---|
| **Fishbone Diagram** (이시카와 다이어그램) | 원인을 카테고리별로 분류 (사람, 프로세스, 도구, 환경) | 원인이 여러 갈래일 때 |
| **Fault Tree Analysis** | 장애를 트리 구조로 분해, 각 분기에 AND/OR 조건 | 복합 장애 분석 |
| **Timeline Analysis** | 시간순으로 이벤트를 나열하여 인과관계 파악 | 장애 경과가 복잡할 때 |
| **Pre-Mortem** | 출시 전 "이미 장애가 났다고 가정"하고 원인을 역추적 | 예방적 분석 |
