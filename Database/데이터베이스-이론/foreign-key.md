# 외래키(Foreign Key)와 참조 무결성

> 최종 업데이트: 2026-05-19 | 기준: SQL 표준 / MySQL 8.x(InnoDB) · PostgreSQL 16

## 개념

**외래키(FK)** 는 한 테이블의 컬럼이 다른 테이블의 **기본키(또는 UNIQUE 키)를 가리키도록** 거는 제약(constraint)이다. FK가 걸려 있으면 DB가 **참조 무결성(referential integrity)** 을 강제한다 — 존재하지 않는 부모를 가리키는 자식 행(고아 레코드)을 만들 수 없고, 자식이 매달린 부모를 함부로 지울 수 없다.

> 비유하자면 "사물함 열쇠와 사물함" 관계. 열쇠(자식, FK)는 반드시 실재하는 사물함(부모, PK)에 대응해야 하고, 누가 쓰고 있는 사물함은 마음대로 철거할 수 없다.

```sql
CREATE TABLE orders (
    id      BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    CONSTRAINT fk_orders_user
        FOREIGN KEY (user_id) REFERENCES users(id)
);
-- users에 없는 user_id로 orders INSERT → 거부됨
```

## 배경/역사

- 외래키와 참조 무결성은 **E. F. Codd의 관계형 모델(1970)** 에서 정식화된 개념으로, 관계형 DB의 핵심 정합성 장치다.
- **SQL-89**에서 FOREIGN KEY 구문이 표준화되고, **SQL-92**에서 `ON DELETE / ON UPDATE` 참조 동작(CASCADE 등)이 추가되었다.
- 즉 FK는 신기술이 아니라 관계형 DB의 토대다. 최근 "FK 안 쓴다" 흐름은 이 개념을 부정하는 게 아니라, **무결성 강제 위치를 DB에서 애플리케이션으로 옮기는** 운영상의 트레이드오프다(아래 별도 섹션).

## 참조 동작 — ON DELETE / ON UPDATE

부모 행이 삭제·변경될 때 자식 행을 어떻게 처리할지 지정한다.

| 옵션 | 동작 |
|---|---|
| `RESTRICT` / `NO ACTION` | 자식이 있으면 부모 삭제·변경 **거부** (기본값) |
| `CASCADE` | 부모 삭제·변경 시 자식도 **함께 삭제·변경** |
| `SET NULL` | 부모 삭제 시 자식 FK 컬럼을 **NULL로** (컬럼이 NULL 허용일 때만) |
| `SET DEFAULT` | 자식 FK를 기본값으로 (지원 DB 제한적) |

```sql
-- 사용자가 지워지면 그 사용자의 주문도 같이 삭제
FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE

-- 카테고리가 지워지면 상품의 category_id만 NULL 처리
FOREIGN KEY (category_id) REFERENCES categories(id) ON DELETE SET NULL
```

> `CASCADE`는 편리하지만 위험하다. 부모 하나를 지웠을 때 연쇄로 수만 행이 삭제되거나, 다단계 캐스케이드로 의도치 않은 대량 삭제가 일어날 수 있다. 삭제 영향 범위를 반드시 확인하고 쓴다.

## 동작과 비용

FK가 걸리면 자식 테이블에 쓰기(INSERT/UPDATE)가 일어날 때마다 DB는 **부모 행의 존재를 확인**하고, 그 부모 행에 **공유 락**을 잡는다. 부모를 지울 때는 반대로 자식을 확인한다. 이것이 무결성을 보장하는 메커니즘이자, 동시에 비용이다.

- **인덱스 필수**: 자식의 FK 컬럼에 인덱스가 없으면 부모 삭제·자식 검증 시 풀스캔이 발생해 치명적으로 느려진다.
  - **MySQL(InnoDB)**: FK 생성 시 자식 컬럼에 인덱스를 **자동 생성**한다.
  - **PostgreSQL**: 자동 생성하지 않는다 → **직접 인덱스를 만들어야** 한다 (가장 흔한 성능 함정).
- **락 경합**: 인기 부모 행(예: 공용 코드 테이블)을 여러 자식이 동시에 참조하면 부모행 락에서 경합·데드락이 생길 수 있다.

## DB별 주의점

| 항목 | 내용 |
|---|---|
| MySQL InnoDB | FK 지원·강제. 자식 FK 컬럼 인덱스 자동 생성 |
| MySQL **파티셔닝 테이블** | **FK 사용 불가** (파티셔닝과 FK 공존 불가) |
| MySQL MyISAM | 문법은 받지만 **FK를 무시**(강제 안 함) — InnoDB 사용 필수 |
| PostgreSQL | FK 지원·강제. 인덱스 **수동** 생성 필요. 옵티마이저가 FK를 최적화에 활용 |
| 무중단 DDL 툴 | `gh-ost`·`pt-online-schema-change`는 FK가 있으면 동작이 까다로워짐 (대규모 운영의 실질적 부담) |

## FK를 걸 것인가 — 설계 트렌드

"요즘은 FK를 안 건다"는 말은 **반은 맞고 반은 과장**이다. 정확히는 **FK 제약을 DB에 안 거는** 선택이며, FK 컬럼·논리적 관계 자체를 없애는 게 아니라 **참조 무결성 강제를 애플리케이션으로 이전**하는 트레이드오프다.

### 제약을 빼는 이유 (대규모·분산)

| 이유 | 설명 |
|---|---|
| MSA / DB 분리 | 서비스별 DB가 분리되면 서비스 경계를 넘는 FK가 **물리적으로 불가능**. 무결성은 이벤트/사가/보상 트랜잭션으로 |
| 샤딩·파티셔닝 | MySQL은 파티셔닝 테이블에 FK 불가. 샤드를 가로지르는 FK도 불가 |
| 쓰기 성능·락 | 쓰기마다 부모 확인 + 부모행 락 → 고동시성에서 경합·데드락, 대량 적재 지연 |
| 스키마 운영 | 무중단 DDL·테이블 드롭·아카이빙·대량 삭제를 FK가 가로막음 |
| CDC/이벤트 파이프라인 | Debezium CDC→Kafka처럼 데이터가 **비동기·최종 일관성**으로 흐르면 강한 FK 가정이 안 맞음 |

### 그래도 FK가 유효한 이유

- **최후의 방어선**: 앱 버그·수동 스크립트·다중 writer가 있어도 고아 레코드가 원천 차단. 앱 검증은 뚫려도 DB 제약은 안 뚫린다.
- 단일 DB·중간 규모에선 FK 오버헤드가 무시할 수준 → 안전을 버릴 이유가 없다(미리 빼는 건 조기 최적화).
- `ON DELETE CASCADE` 등 편의, 스키마가 곧 문서, 일부 옵티마이저는 FK를 최적화에 활용.

### 판단 기준

| 상황 | 권장 |
|---|---|
| 단일 DB 모놀리식, 일반 CRUD, 중간 트래픽 | **FK 건다** (정석, 안전) |
| 단일 DB·초고동시성 쓰기·대량 배치 | 핫 테이블만 선택적으로 FK 제거 + 앱/배치 검증 |
| MSA·서비스별 DB·샤딩·CDC/이벤트 파이프라인 | **FK 제약 안 검** — 무결성을 앱·이벤트·정합성 배치로 이전 |

### FK를 빼더라도 반드시

1. FK **컬럼과 논리적 관계는 유지**한다 (관계를 없애는 게 아님)
2. 그 컬럼에 **인덱스를 수동으로 직접 생성**한다 (FK가 없으면 자동 인덱스도 없음 → 조인·조회 성능 추락)
3. 무결성 검증 로직을 **애플리케이션 계층에 명시적으로** 둔다
4. 주기적 **정합성 점검 배치**로 고아 레코드를 탐지·보정한다

> 요약: "FK 안 쓴다"는 대규모·분산에서 DB 제약을 앱으로 옮기는 트레이드오프일 뿐, 관계형 설계를 포기하는 게 아니다. 규모가 그 수준이 아니면 FK는 그대로 쓰는 것이 맞다.
