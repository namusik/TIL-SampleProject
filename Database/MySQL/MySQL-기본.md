# MySQL 기본

> 최종 업데이트: 2026-05-20 | 기준: MySQL 8.x (InnoDB)

## 개념

**MySQL**은 가장 널리 쓰이는 **오픈소스 관계형 데이터베이스(RDBMS)** 다. 클라이언트-서버 구조로 동작하며, SQL로 데이터를 정의·조작한다. 가볍고 빠르며 웹 서비스 백엔드의 사실상 표준 중 하나다.

> 비유하자면 "표준 규격의 대형 창고". 물건(데이터)을 칸(테이블)에 규칙대로 넣고, 송장(SQL)으로 입출고한다. 창고를 운영하는 방식(스토리지 엔진)은 교체할 수 있다.

## 배경/역사

- **개발자**: Michael "Monty" Widenius 등, 스웨덴 **MySQL AB**(1995). 이름은 Monty의 딸 "My"에서 유래.
- **소유권 이동**: 2008년 Sun Microsystems 인수 → 2010년 **Oracle**이 Sun을 인수하며 MySQL도 Oracle 소유.
- **MariaDB 분기**: Oracle 인수 후 라이선스 우려로 창립자가 **MariaDB**를 포크(드롭인 호환 지향). 현재 둘은 일부 기능이 갈라짐.
- **라이선스**: GPL + 상용 듀얼 라이선스.
- **주요 버전**: 5.7 → **8.0**(2018, 대규모 개편) → 8.x 연속 릴리스 체계.

## 아키텍처 — 3계층

MySQL을 다른 DBMS와 구분 짓는 핵심은 **스토리지 엔진이 교체 가능(pluggable)** 하다는 설계다. 이는 PostgreSQL·Oracle 등 단일 엔진 구조와 가장 다른 지점이다.

| 계층 | 역할 |
|---|---|
| 연결 계층 | 커넥션·인증·스레드 관리 |
| SQL 계층 | 파서 → 옵티마이저 → 실행기 |
| 스토리지 엔진 계층 | 실제 데이터 저장·인덱스·락·트랜잭션 처리 (InnoDB/MyISAM/Memory…) |

같은 SQL이라도 어떤 엔진을 쓰느냐에 따라 트랜잭션·락·FK 지원 여부가 달라진다.

## 스토리지 엔진

| 엔진 | 트랜잭션 | 락 단위 | FK | 용도 |
|---|---|---|---|---|
| **InnoDB** (기본) | ✓ | 행(row) | ✓ | 거의 모든 운영 테이블 — MVCC, 크래시 복구 |
| MyISAM | ✗ | 테이블 | ✗ | 레거시·읽기 전용, 신규 사용 비권장 |
| MEMORY | ✗ | 테이블 | ✗ | 임시·캐시성 데이터(재시작 시 소실) |

> 특별한 이유가 없으면 **InnoDB**를 쓴다. 트랜잭션·행 단위 락·외래키·MVCC·크래시 복구를 모두 지원한다. PK는 InnoDB의 **클러스터링 인덱스**가 된다([Auto Increment](../데이터베이스-이론/auto-increment.md) 참조).

## 다른 DBMS와 무엇이 다른가

| 비교 항목 | MySQL | PostgreSQL | Oracle |
|---|---|---|---|
| 성격 | 단순·빠름·운영 쉬움 | 표준 준수·기능 풍부(객체관계형) | 엔터프라이즈·고가 상용 |
| 스토리지 엔진 | **교체 가능(pluggable)** | 단일 엔진 | 단일 엔진 |
| 기본 격리 수준 | **REPEATABLE READ** | READ COMMITTED | READ COMMITTED |
| 복잡 쿼리·분석 | 상대적으로 약함 | 강함(CTE·윈도우·다양한 인덱스 타입) | 매우 강함 |
| 복제 | **단순·성숙**(생태계 방대) | 강력하나 설정 복잡 | 고가 옵션 |
| 라이선스/비용 | 오픈소스(듀얼) | 완전 오픈소스(PostgreSQL 라이선스) | 고가 상용 |

MySQL을 특징짓는 성향과 그에 따른 함정:

- **단순함·읽기 성능·운영 편의 우선** 설계. 그래서 대규모 웹 트래픽·복제 기반 스케일아웃에 강하지만, 무거운 분석/복잡 쿼리는 PostgreSQL·Oracle보다 약하다.
- **기본 격리 수준이 REPEATABLE READ** — 대부분의 DBMS(PostgreSQL/Oracle/SQL Server)가 READ COMMITTED인 것과 달라, 다른 DB 경험을 그대로 적용하면 팬텀·갭락 동작에서 혼동이 생긴다. ([Transaction Isolation Level](Transaction-Isolation-Level.md))
- **문자셋 `utf8`의 함정**: MySQL의 `utf8`은 최대 3바이트라 이모지·일부 한자가 깨진다. 반드시 4바이트 완전 UTF-8인 **`utf8mb4`** 를 써야 한다(8.0부터 서버 기본값이 `utf8mb4`, 5.7 이하는 `latin1`). 이는 다른 DBMS에 없는 MySQL 특유의 주의점이다. ([Charset 확인](Charset-확인.md))
- **복제 생태계가 압도적**: 비동기/반동기 복제, 다양한 미들웨어, 클라우드 매니지드(RDS/Aurora) 지원이 가장 성숙 — 스케일아웃 표준 선택지가 되는 핵심 이유.

## 장점과 한계

| 장점 | 한계 |
|---|---|
| 가볍고 설치·운영이 쉬움(낮은 진입장벽) | 복잡 쿼리·분석·고급 SQL은 상대적으로 약함 |
| 단순 읽기/쓰기 성능 우수 | 표준 SQL 준수도가 PostgreSQL보다 낮음(이식성 ↓) |
| 방대한 레퍼런스·커뮤니티·운영 노하우 | 일부 기능이 스토리지 엔진별로 달라 혼동 가능 |
| 복제·클라우드 매니지드 생태계 성숙 | 매니지드(Aurora 등)에서 일부 동작이 커뮤니티판과 다름 |
| 오픈소스(비용 부담 작음) | Oracle 소유 → 일부 기능/라이선스 정책 불확실성(→ MariaDB 분기 배경) |

> 정리: **"단순·빠름·운영 쉬움 + 복제로 스케일아웃"** 이 MySQL의 정체성. 트랜잭션·복잡 분석·표준 준수가 핵심이면 PostgreSQL/Oracle을, 대규모 웹 트래픽과 운영 편의가 우선이면 MySQL을 선택하는 것이 일반적 판단이다.

## 5.7 → 8.0 주요 변경

| 항목 | 변화 |
|---|---|
| 기본 문자셋 | `latin1` → **`utf8mb4`** |
| CTE / 윈도우 함수 | 8.0에서 추가 (`WITH`, `ROW_NUMBER() OVER` 등) — 복잡 쿼리 약점 일부 보완 |
| 원자적 DDL | DDL이 원자적으로 처리(중단 시 부분 적용 방지) |
| auto_increment | 카운터를 redo log에 영속화(커뮤니티 한정 — **Aurora는 예외**, [Auto Increment](../데이터베이스-이론/auto-increment.md) 참조) |
| 쿼리 캐시 | **제거됨** (8.0) |
| 기본 인증 플러그인 | `caching_sha2_password`로 변경(구 클라이언트 호환 주의) |
