# Vector Database (벡터 데이터베이스)

> 최종 업데이트: 2026-03-27

## 개념

**벡터(숫자 배열)를 저장하고, 유사한 벡터를 빠르게 검색**하는 데 특화된 데이터베이스.

- 도서관에 비유하면: 일반 DB는 책 제목·저자로 정확히 검색하는 도서관 시스템이고, 벡터DB는 **"이 책과 비슷한 분위기의 책 추천해줘"** 라는 요청을 처리할 수 있는 시스템
- 기존 DB가 **정확 일치(exact match)** 에 강하다면, 벡터DB는 **유사도 검색(similarity search)** 에 강함

## 벡터(Embedding)란

텍스트, 이미지, 오디오 등의 데이터를 AI 모델(임베딩 모델)이 **숫자 배열(벡터)** 로 변환한 것.

```
"오늘 날씨가 좋다" → [0.12, -0.34, 0.56, 0.78, ...]   (수백~수천 차원)
"날씨가 화창하다"  → [0.11, -0.33, 0.55, 0.80, ...]   ← 의미가 비슷하면 벡터도 가까움
"주식이 올랐다"    → [0.91, 0.22, -0.45, 0.10, ...]   ← 의미가 다르면 벡터도 멀어짐
```

- 의미가 비슷한 데이터는 벡터 공간에서 **가까운 위치**에 놓임
- 이 거리를 계산해서 "비슷한 것"을 찾는 것이 벡터 검색의 핵심

## 왜 일반 DB로는 안 되는가

| 구분 | 일반 RDBMS | 벡터 DB |
|------|-----------|---------|
| 검색 방식 | WHERE name = 'foo' (정확 일치) | "이것과 비슷한 것" (유사도) |
| 데이터 | 정형 데이터 (문자열, 숫자) | 고차원 벡터 (수백~수천 차원) |
| 인덱스 | B-Tree, Hash | ANN 인덱스 (HNSW, IVF 등) |
| 성능 | 고차원 벡터 유사도 검색에 부적합 | 수백만 벡터에서 밀리초 단위 검색 |

- 벡터 간 유사도를 구하려면 모든 벡터와 거리 계산을 해야 하는데, 데이터가 많아지면 불가능 → 벡터DB는 **ANN(Approximate Nearest Neighbor)** 알고리즘으로 근사 검색하여 속도를 확보

## 유사도 측정 방식

벡터 간 "얼마나 가까운가"를 측정하는 방법. 쇼핑몰에서 상품 추천할 때 "이 상품과 취향이 얼마나 비슷한가"를 수치로 계산하는 것과 같음.

| 방식 | 설명 | 특징 |
|------|------|------|
| Cosine Similarity | 벡터 간 각도 기반 | 방향(의미)이 비슷한지 비교. 텍스트 검색에서 가장 많이 사용 |
| Euclidean Distance | 벡터 간 직선 거리 | 절대적 거리. 값의 크기가 중요할 때 |
| Dot Product | 벡터 내적 | 방향 + 크기 모두 반영 |

## ANN (Approximate Nearest Neighbor) 인덱스

정확한 최근접 이웃 대신 **근사치를 빠르게** 찾는 알고리즘. 100% 정확도를 약간 포기하고 속도를 수천 배 높이는 트레이드오프.

| 알고리즘 | 방식 | 특징 |
|----------|------|------|
| HNSW | 그래프 기반 탐색 | 검색 속도·정확도 밸런스 우수. 가장 널리 사용 |
| IVF | 클러스터링 후 해당 클러스터만 검색 | 대규모 데이터에 효율적 |
| PQ (Product Quantization) | 벡터를 압축하여 메모리 절약 | 메모리 제한 환경에 적합 |

## 주요 벡터 DB 제품

| 제품 | 특징 |
|------|------|
| **Pinecone** | 완전 관리형 SaaS. 설정 최소화, 빠른 도입 |
| **Milvus** | 오픈소스. 대규모 데이터 처리에 강함 (Zilliz가 클라우드 서비스 제공) |
| **Weaviate** | 오픈소스. 자체 임베딩 모듈 내장 |
| **Qdrant** | 오픈소스. Rust 기반으로 고성능 |
| **Chroma** | 오픈소스. 경량, 로컬 개발에 간편 |
| **pgvector** | PostgreSQL 확장. 기존 RDB에 벡터 검색 추가 |

- 새로 벡터 전용 DB를 도입하기 부담스러우면 **pgvector**로 기존 PostgreSQL에 벡터 검색을 추가하는 방식도 가능

## 주요 활용 사례

| 사례 | 설명 |
|------|------|
| RAG (검색 증강 생성) | LLM이 답변 전 관련 문서를 벡터DB에서 검색하여 정확도 향상 |
| 시맨틱 검색 | 키워드가 아닌 의미 기반 검색 ("저렴한 숙소" → "가성비 좋은 호텔"도 매칭) |
| 추천 시스템 | 유저 행동·상품을 벡터화하여 유사 상품/콘텐츠 추천 |
| 이미지 검색 | 이미지를 벡터화하여 유사 이미지 검색 |
| 이상 탐지 | 정상 패턴 벡터와 거리가 먼 데이터를 이상치로 판별 |

## RAG에서의 벡터DB 흐름

LLM 기반 서비스에서 가장 대표적인 활용 패턴.

```
[저장 단계]
문서 → 청크 분할 → 임베딩 모델 → 벡터 → 벡터DB에 저장

[검색 단계]
사용자 질문 → 임베딩 모델 → 질문 벡터
    → 벡터DB에서 유사 문서 검색
    → 검색된 문서 + 질문을 LLM에 전달
    → LLM이 문서 기반으로 답변 생성
```
