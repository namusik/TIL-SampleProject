# 테스트 코드 개론

## 왜 테스트를 해야 하는가?

### 1. 버그 조기 발견
코드가 배포되기 전에 결함을 찾아낸다. 배포 후 발견되는 버그는 수정 비용이 기하급수적으로 증가한다.

### 2. 리팩토링에 대한 안전망
테스트가 있으면 기존 기능이 깨지지 않았음을 빠르게 확인할 수 있다.
테스트 없이 리팩토링하는 것은 줄 없이 번지점프를 하는 것과 같다.

### 3. 문서로서의 역할
테스트 코드는 "이 코드가 어떻게 동작해야 하는지"를 보여주는 살아있는 문서다.
주석과 달리 코드와 동기화가 깨지지 않는다 (깨지면 테스트가 실패하므로).

### 4. 설계 품질 향상
테스트하기 어려운 코드 = 설계가 나쁜 코드인 경우가 많다.
테스트를 작성하면 자연스럽게 느슨한 결합(loose coupling)과 높은 응집도(high cohesion)를 추구하게 된다.

### 5. 배포 자신감
CI/CD 파이프라인에서 자동화된 테스트가 통과해야 배포되므로, 안정적인 릴리스가 가능하다.

## 테스트의 종류

### 테스트 피라미드 (Test Pyramid)

```
        /  E2E   \          ← 느리고, 비용 높음, 적게 작성
       /----------\
      / Integration \       ← 중간 속도, 중간 비용
     /----------------\
    /    Unit Tests     \   ← 빠르고, 비용 낮음, 많이 작성
   /____________________\
```

아래로 갈수록 테스트가 많아야 하고, 위로 갈수록 적어야 한다.

### 1. 단위 테스트 (Unit Test)
가장 작은 단위(메서드, 클래스)를 독립적으로 검증한다.

- 외부 의존성 없이 실행 (DB, 네트워크 등 불필요)
- 실행 속도가 매우 빠름 (ms 단위)
- 도구: JUnit5, Mockito, AssertJ

```java
@Test
void 주문금액이_5만원_이상이면_배송비_무료() {
    Order order = new Order(50000);

    int shippingFee = order.calculateShippingFee();

    assertThat(shippingFee).isZero();
}
```

### 2. 통합 테스트 (Integration Test)
여러 모듈이 함께 동작할 때 올바르게 작동하는지 검증한다.

- 실제 DB, 외부 API 등과 연동
- 단위 테스트보다 느리지만, 모듈 간 상호작용 검증 가능
- 도구: @SpringBootTest, Testcontainers, LocalStack

```java
@SpringBootTest
@Transactional
class OrderServiceIntegrationTest {
    @Autowired OrderService orderService;

    @Test
    void 주문_생성시_재고가_차감된다() {
        OrderRequest request = new OrderRequest("상품A", 2);

        orderService.createOrder(request);

        assertThat(stockRepository.getStock("상품A")).isEqualTo(8);
    }
}
```

### 3. 인수 테스트 (Acceptance Test)
사용자 시나리오 관점에서 시스템 전체를 검증한다.

- 비즈니스 요구사항 기반
- API 엔드포인트 호출 → 응답 검증 형태가 많음
- 도구: RestAssured, MockMvc

```java
@Test
void 회원가입_후_로그인_성공() {
    // 회원가입
    given()
        .body(new SignUpRequest("user@test.com", "password123"))
    .when()
        .post("/api/members")
    .then()
        .statusCode(201);

    // 로그인
    given()
        .body(new LoginRequest("user@test.com", "password123"))
    .when()
        .post("/api/login")
    .then()
        .statusCode(200)
        .body("token", notNullValue());
}
```

### 4. E2E 테스트 (End-to-End Test)
실제 사용자 환경과 동일한 조건에서 전체 시스템을 검증한다.

- 브라우저 자동화 포함 (프론트엔드 + 백엔드 + DB)
- 가장 느리고, 유지보수 비용이 높음
- 도구: Selenium, Cypress, Playwright

## 무엇을 테스트해야 하는가?

### 테스트해야 하는 것

| 대상 | 예시 |
|------|------|
| **비즈니스 로직** | 할인 계산, 주문 상태 변경, 권한 검증 |
| **경계값(Edge Case)** | 0, null, 빈 문자열, 최대/최소값 |
| **예외 처리** | 잘못된 입력, 존재하지 않는 리소스 접근 |
| **상태 변화** | 주문 생성 → 재고 차감, 결제 → 포인트 적립 |
| **외부 연동 계약** | 외부 API 호출 시 요청/응답 형식 |

### 테스트하지 않아도 되는 것

| 대상 | 이유 |
|------|------|
| **프레임워크/라이브러리 자체** | Spring의 DI, JPA의 findById 등은 이미 검증됨 |
| **단순 getter/setter** | 로직이 없는 코드를 테스트하는 것은 낭비 |
| **외부 시스템 그 자체** | 외부 API가 정상인지는 우리가 테스트할 영역이 아님 |
| **private 메서드** | public 메서드를 통해 간접적으로 검증 |

### 핵심 원칙: "변경 가능성이 있는 비즈니스 로직"에 집중

테스트 커버리지 100%가 목표가 아니다. 핵심 비즈니스 로직 80% 커버가 getter/setter 포함 100% 커버보다 가치있다.

## 좋은 테스트의 특성

### FIRST 원칙
| 원칙 | 설명 |
|------|------|
| **F**ast | 빠르게 실행되어야 한다 |
| **I**solated | 다른 테스트에 의존하지 않아야 한다 |
| **R**epeatable | 언제 실행해도 같은 결과 |
| **S**elf-validating | 성공/실패가 자동으로 판별 |
| **T**imely | 프로덕션 코드와 함께 작성 |

### Given-When-Then 패턴
```
Given : 테스트의 사전 조건 (데이터 준비)
When  : 테스트 대상 행위 실행
Then  : 결과 검증
```

### 하나의 테스트, 하나의 검증
한 테스트에서 여러 시나리오를 검증하면 실패 원인 파악이 어려워진다.

## 테스트 관련 주요 개념

### Test Double
실제 객체 대신 테스트에 사용하는 대체 객체

| 종류 | 설명 |
|------|------|
| **Stub** | 미리 정해진 값을 반환 |
| **Mock** | 호출 여부와 방식을 검증 |
| **Spy** | 실제 객체를 감싸서 호출 기록 |
| **Fake** | 실제 동작하지만 단순화된 구현 (ex. InMemoryRepository) |
| **Dummy** | 전달만 되고 실제 사용되지 않는 객체 |

### SUT와 DOC
- **SUT** (System Under Test): 테스트 대상
- **DOC** (Depended On Component): SUT가 의존하는 컴포넌트

### Code Coverage
테스트가 소스 코드를 얼마나 실행했는지를 나타내는 지표.

- Line Coverage: 실행된 코드 라인 비율
- Branch Coverage: 분기(if/else)를 얼마나 커버했는지
- 도구: JaCoCo, Cobertura

> 커버리지가 높다고 좋은 테스트는 아니다. 하지만 커버리지가 낮으면 테스트가 부족한 것은 확실하다.

## 테스트 전략 요약

```
1. 단위 테스트로 핵심 비즈니스 로직을 촘촘하게 검증
2. 통합 테스트로 모듈 간 상호작용과 DB 연동을 확인
3. 인수 테스트로 주요 사용자 시나리오를 보장
4. CI/CD 파이프라인에 테스트를 자동화
```

테스트는 "보험"이다. 당장은 비용이지만, 장기적으로는 개발 속도를 높여준다.
