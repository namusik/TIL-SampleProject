# C/C++ 빌드 파일 확장자 (.c / .o / .a / .so / .in)

> 최종 업데이트: 2026-04-22 | 기준: Linux/macOS/Windows 공통 C·C++ 빌드 관행

## 개념

C/C++ 프로젝트를 빌드하면 소스·중간물·산출물·템플릿 등 **여러 확장자의 파일**이 등장한다. 크게 **입력(소스)**, **중간물**, **산출물(라이브러리/실행파일)**, **템플릿** 네 갈래로 나뉘고, 각각이 빌드 파이프라인의 다른 단계에 위치한다.

> 비유하자면 공장의 자재·반제품·완제품·설계도 같은 역할. `.c`는 재료, `.o`는 반제품, `.a`/`.so`는 완제품, `.in`은 설계 템플릿.

## 빌드 파이프라인 속 위치

```
[입력]                  [중간물]            [산출물]
.c / .cpp / .h  ──컴파일──►  .o  ──아카이브/링크──►  .a  (정적 라이브러리)
                                                 ──►  .so (동적 라이브러리)
                                                 ──►  실행 파일 (확장자 없음)

[템플릿]
.in            ──configure──►  실제 파일 (Makefile, config.h, xxx.pc 등)
```

## 전체 확장자 표

| 확장자 | 풀네임 | 종류 | 설명 |
|--------|--------|------|------|
| **`.c`** | C source | 입력 | C 소스 코드 |
| **`.cpp`**, `.cc`, `.cxx` | C++ source | 입력 | C++ 소스 코드 |
| **`.h`** | Header | 입력 | 함수·타입 선언이 담긴 헤더 |
| **`.hpp`** | C++ Header | 입력 | C++ 헤더 |
| **`.o`** | **Object** | 중간물 | 컴파일된 기계어 — 아직 링크 전 |
| **`.a`** | **Archive** (static library) | 산출물 | **정적 라이브러리** — 빌드 시점에 실행 파일에 통째로 포함 |
| **`.so`** | **Shared Object** | 산출물 | **동적 라이브러리** — 런타임에 OS가 로드 (Linux) |
| **`.dylib`** | Dynamic Library | 산출물 | 동적 라이브러리 (**macOS**) |
| **`.dll`** | Dynamic Link Library | 산출물 | 동적 라이브러리 (**Windows**) |
| **`.lib`** | Library | 산출물 | 정적 라이브러리 또는 DLL 임포트 (**Windows**) |
| **`.la`** | Libtool Archive | 메타 | libtool이 관리하는 메타 정보 (실제 코드 아님) |
| **`.in`** | Input (template) | 템플릿 | `configure`가 치환할 **템플릿** |
| **`.am`** | Automake | 원본 | 개발자가 작성하는 automake 입력 |
| **`.pc`** | pkg-config | 메타 | 라이브러리 메타정보 (include/lib 경로) |
| **`.m4`** | Macro (m4) | 원본 | autoconf 매크로 정의 |

## 핵심 파일 깊게 보기

### `.o` — 오브젝트 파일 (중간물)

컴파일러가 **`.c` → 기계어**로 변환한 결과. 아직 다른 `.o`나 라이브러리와 **링크되지 않은 상태**라 단독 실행 불가.

```sh
cc -c foo.c -o foo.o          # 컴파일만 (링크 X)
```

- 여러 `.o`를 모아 링크하면 실행 파일 또는 라이브러리가 됨
- 거의 항상 빌드 끝나면 지워짐 (`make clean`)

### `.a` — 정적 라이브러리 (Static Library)

여러 `.o` 파일을 **하나로 묶은 아카이브**. 이름 그대로 "archive".

```sh
ar cr libfoo.a foo.o bar.o    # .o들을 묶음
cc main.c libfoo.a -o app     # 빌드 시점에 실행 파일 안에 포함됨
```

- **빌드 타임에 실행 파일에 통째로 복사**됨
- 장점: 배포 시 라이브러리 따로 필요 없음 (단일 실행 파일)
- 단점: 실행 파일 크기 증가, 라이브러리 업데이트 시 재컴파일 필요

### `.so` — 동적/공유 라이브러리 (Shared Object)

**런타임에 OS가 로드**하는 라이브러리. 여러 프로그램이 메모리에 같은 `.so`를 공유.

```sh
cc -shared -fPIC foo.c -o libfoo.so
cc main.c -L. -lfoo -o app    # -lfoo는 libfoo.so를 찾음
./app                          # 실행 시 libfoo.so가 런타임에 로드됨
```

- 실행 시 `ldd app` 로 의존하는 `.so` 확인 가능
- 장점: 메모리 공유(효율↑), 라이브러리만 교체하면 앱 재빌드 불필요
- 단점: 배포 시 `.so`도 같이 가야 함, 경로 문제(`LD_LIBRARY_PATH`)

#### `.so`의 버전 심볼릭 링크 관례

같은 라이브러리의 다른 버전을 동시에 유지하기 위한 표준 구조.

```
libfoo.so        → libfoo.so.1       (링커용 심볼릭 링크)
libfoo.so.1      → libfoo.so.1.2.3   (런타임용 심볼릭 링크, SONAME)
libfoo.so.1.2.3                      (실제 파일)
```

- 빌드 시 `-lfoo`는 `libfoo.so`(심볼릭 링크)를 찾음
- 런타임엔 SONAME(`libfoo.so.1`)을 따라 실제 파일로 연결
- **메이저 버전이 다르면 ABI 호환 안 됨** — 그래서 `.so.1`과 `.so.2`가 공존 가능

### `.a` vs `.so` 비교

| 항목 | `.a` (정적) | `.so` (동적) |
|------|------------|-------------|
| 링크 시점 | 빌드 타임 | 런타임 |
| 실행 파일 크기 | 큼 | 작음 |
| 메모리 사용 | 프로세스마다 복사 | 여러 프로세스 공유 |
| 배포 | 단일 파일 | `.so` 함께 배포 |
| 업데이트 | 재빌드 필요 | `.so`만 교체 |
| 대표 용도 | 임베디드, 정적 빌드, Go 바이너리 | 일반 서버/데스크톱 앱 |

### `.in` — 템플릿 파일

`configure`가 플레이스홀더를 실제 값으로 치환해 실제 파일을 생성하는 **원본 템플릿**.

| 템플릿 | 치환 후 |
|--------|--------|
| `Makefile.in` | `Makefile` |
| `config.h.in` | `config.h` |
| `sqlite3.pc.in` | `sqlite3.pc` |

내부에 `@CC@`, `@CFLAGS@`, `@prefix@` 같은 플레이스홀더가 박혀 있고, `configure` 실행 시 시스템 환경에 맞는 값으로 채워짐.

자세한 건 [소스 빌드 (configure make).md](소스-빌드-%28configure-make%29.md) 참고.

## 플랫폼별 라이브러리 확장자

같은 개념을 OS마다 다른 확장자로 부른다.

| 구분 | Linux | macOS | Windows |
|------|-------|-------|---------|
| 정적 라이브러리 | `.a` | `.a` | `.lib` |
| 동적 라이브러리 | `.so` | `.dylib` | `.dll` |
| 실행 파일 | (확장자 없음) | (없음) | `.exe` |
| 링크용 임포트 | — | — | `.lib` (DLL의 심볼 정보) |

> Windows의 `.lib`는 **두 가지**다 — 진짜 정적 라이브러리일 수도, DLL과 짝을 이루는 **임포트 라이브러리**일 수도 있음.

## 부수적인 파일들

### `.pc` — pkg-config

라이브러리 연동용 **메타파일**. 컴파일러에게 "이 라이브러리 쓰려면 어떤 플래그가 필요하냐"를 알려준다.

```sh
pkg-config --cflags --libs sqlite3
# -I/usr/include -lsqlite3

# Makefile에서 사용
CFLAGS += $(shell pkg-config --cflags sqlite3)
LDFLAGS += $(shell pkg-config --libs sqlite3)
```

### `.la` — Libtool Archive

libtool이 관리하는 메타파일. "이 라이브러리는 어떤 `.so`에 의존한다"는 정보.

- 실제 코드/바이너리 아님 — 텍스트 파일
- 현대 빌드는 pkg-config로 대체되는 추세

### `.m4` — Autoconf 매크로

`configure.ac`(또는 `configure.in`)에서 재사용되는 매크로 정의.

### 빌드 로그/상태 파일

- **`config.log`** — `configure` 실행 상세 로그. 실패 시 필수 디버깅 자료
- **`config.status`** — `configure` 재실행용 스크립트 (같은 옵션으로 다시 돌릴 때)
- **`Makefile.am`** — 개발자가 작성하는 automake 원본

## 명령어별 등장 파일

| 명령어 | 입력 | 출력 |
|--------|------|------|
| `cc -c foo.c` | `.c`, `.h` | `.o` |
| `ar cr libfoo.a *.o` | `.o` | `.a` |
| `cc -shared -o libfoo.so *.o` | `.o` | `.so` |
| `cc main.c -lfoo -o app` | `.c`, `.a`/`.so` | 실행 파일 |
| `./configure` | `.in`, `configure` | `Makefile`, `config.h` 등 |
| `make` | `Makefile`, `.c`, `.h` | `.o`, `.a`, `.so`, 실행 파일 |

## 실무 팁

- **실행 파일이 안 돌아갈 때** — `ldd app`로 의존 `.so`가 `not found` 인지 확인
- **`.so` 경로 문제** — `LD_LIBRARY_PATH` 환경변수 또는 `/etc/ld.so.conf.d/*.conf` + `sudo ldconfig`
- **헤더 못 찾는다고 할 때** — `-I/경로` 추가 (또는 `-dev`/`-devel` 패키지 설치)
- **`.a` vs `.so` 둘 다 있을 때** — 기본은 `.so`를 우선. 강제로 정적 링크하려면 `-static` 플래그
- **pkg-config가 라이브러리를 못 찾음** — `PKG_CONFIG_PATH` 확인
- **임베디드/컨테이너 초경량 바이너리** — `.a`로 정적 링크하면 `.so` 없이 단독 실행 가능

## 요약

- **`.c/.h`** (재료) → **`.o`** (반제품) → **`.a`** (정적 통짜) 또는 **`.so`** (동적 공유)
- **`.a`는 빌드 타임에 합쳐지고**, **`.so`는 런타임에 로드됨** — 이 두 개가 핵심
- **`.in`** 은 템플릿 — `configure`가 실제 파일 생성
- macOS는 `.dylib`, Windows는 `.dll/.lib` — 이름만 다를 뿐 개념은 동일
