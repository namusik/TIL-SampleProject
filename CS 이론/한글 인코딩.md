# 한글 인코딩

## 1. 인코딩 기본 개념

- 컴퓨터는 모든 것을 숫자로 처리한다. 문자를 저장하려면 문자를 숫자로 바꿔야 한다.
- **인코딩**: 문자를 숫자로 바꾸는 규칙이나 방식

- **문자 집합 (Character Set)**
  - 표현해야 할 문자를 정하고 순서를 지정한 것
  - 예: 가 ~ 힣, A ~ Z

- **코드화된 문자 집합 (CCS, Coded Character Set)**
  - 문자 집합의 각 문자에 고유한 숫자(코드 포인트)를 부여한 것
  - 예: 가 → 10001, 각 → 10002

- **인코딩 방식 (CES, Character Encoding Scheme)**
  - 코드 포인트를 실제 바이트(옥텟, 8비트 단위)로 변환하는 방식
  - 예: UTF-8, UTF-16, EUC-KR

---

## 2. ASCII와 확장

- **ASCII** (American Standard Code for Information Interchange)
  - 0x00 ~ 0x7F까지 총 **128개** 문자 (제어문자, 특수 문자, 숫자, 알파벳)
  - 7비트로 표현되며, 영문 기반의 표준 문자 인코딩
  - ü 같은 유럽어 문자는 표현 불가

- **확장 ASCII** (Extended ASCII)
  - 0x80 ~ 0xFF를 추가하여 총 **256개** 문자 표현 가능
  - ISO-8859 시리즈로 유럽 표준안 제정

---

## 3. 한글 인코딩의 역사

- **조합형**
  - 한글의 제자 원리에 기반하여 초성, 중성, 종성에 각각 코드를 할당
  - 모든 한글 조합을 표현할 수 있다는 장점
  - N바이트 조합형, 3바이트 조합형, 2바이트 조합형

- **완성형**
  - 완성된 글자(가, 한 등)에 코드를 할당하는 방식
  - KS C 5601로 한글 표준안 채택
  - 2,350자만 수록하여 "똠방각하"의 '똠' 등 일부 글자 표현 불가 문제
  - 이후 확장 완성형(CP949)에서 11,172자로 보완

- **EUC-KR**: 완성형 기반, 2,350자의 한글
- **CP949 (MS949)**: EUC-KR의 확장, 11,172자의 한글. Windows 환경 기본 인코딩

## 4. 유니코드

- 전 세계 모든 문자 집합을 하나로 통합한 국제 표준
- 기존 인코딩들이 서로 호환되지 않는 문제를 해결하기 위해 탄생
- ISO 10646 표준의 UCS(Universal Character Set) 기반
- 유니코드 2.0부터 한글 11,172자 포함
- **코드 포인트**: 각 문자에 부여된 고유 번호. `U+` 접두사로 표기 (예: `U+AC00` = '가')

## 5. UTF-8 / UTF-16

### UTF-8
- Universal Coded Character Set + Transformation Format – 8-bit
- **가변 길이**: 1~4바이트
  - ASCII(영문, 숫자): 1바이트
  - 한글, 한자: **3바이트**
  - 이모지 등 보조 평면: 4바이트
- ASCII와 완전 호환되어 웹 표준 인코딩으로 가장 널리 사용됨

### UTF-16
- 2바이트 또는 4바이트(서로게이트 페어) 사용
- 한글: **2바이트**로 UTF-8(3바이트)보다 효율적
- BOM(Byte Order Mark)으로 엔디언 구분: BE(Big Endian) / LE(Little Endian)

## 6. Java에서의 한글 인코딩

- Java의 `String`은 내부적으로 **UTF-16** 인코딩으로 문자열을 저장한다.
  - Java 9+: **Compact Strings** 도입으로 Latin-1 범위 문자만 있으면 1바이트/문자로 저장하여 메모리 절약
- 문자열 입출력 시에만 사용자가 지정한 인코딩(UTF-8, EUC-KR 등)으로 변환한다.
- JVM 기본 인코딩(`file.encoding`)은 JVM 로딩 시에만 초기화되므로, 코드 중간에 프로퍼티를 변경해도 반영되지 않는다.
