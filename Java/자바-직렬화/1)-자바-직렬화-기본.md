# 자바 직렬화 (Serialization)

> 최종 업데이트: 2026-04-28 | Java 21 LTS 기준

## 개념

자바 직렬화는 **JVM 메모리에 있는 객체를 외부에서 사용할 수 있는 형태(바이트, 문자열 등)로 변환하는 기술**이다. 반대로 외부 데이터를 다시 객체로 복원하는 것은 **역직렬화(Deserialization)** 라고 한다.

> 비유: 한국에서 만든 가구를 미국으로 보내려면 분해해서 박스에 담아야 한다(직렬화). 도착한 박스를 다시 조립하면 원래 가구가 된다(역직렬화).

## 왜 필요한가

JVM 안에서만 살던 객체를 바깥 세상으로 내보내거나, 다시 들여올 때 필요하다. 객체의 **참조(주소)는 JVM 안에서만 유효**하기 때문에 그대로 내보낼 수 없고, 외부가 이해할 수 있는 형태로 변환해야 한다.

| 상황 | 예시 |
|---|---|
| 네트워크 전송 | REST API 응답, RPC 호출, 메시지 큐 |
| 파일/DB 저장 | 객체 상태를 영속화, 세션 저장 |
| 프로세스 간 통신 | JVM 사이, 또는 다른 언어와 데이터 교환 |
| 캐시 저장 | Redis, Memcached에 객체 저장 |

## 직렬화 방식 비교

| 방식 | 포맷 | 장점 | 단점 | 주 용도 |
|---|---|---|---|---|
| Java Native (Serializable) | Binary | JDK 내장, 객체 그래프 그대로 보존 | 보안 취약, JVM 종속, 버전 호환 어려움 | 거의 사용 안 함 |
| JSON (Jackson, Gson) | Text | 사람이 읽기 좋음, 언어 독립적 | 크기 큼, 타입 정보 약함 | REST API, 일반 통신 |
| XML (JAXB) | Text | 스키마 강제, 검증 가능 | 장황함, 파싱 느림 | 레거시, SOAP |
| Protocol Buffers | Binary | 작고 빠름, 스키마 강제 | .proto 파일 필요 | gRPC, 고성능 통신 |
| Avro | Binary | 스키마 진화 강력 | 도구 의존성 | 빅데이터, Kafka |
| Kryo | Binary | 빠름, 자바 친화적 | 자바 종속 | 캐시, Spark 내부 |

> **현대 백엔드 기준 사실상 표준은 JSON(Jackson)**. 성능이 중요한 마이크로서비스 간 통신은 Protocol Buffers, 빅데이터는 Avro를 쓴다.

## Java Native 직렬화는 왜 안 쓰나

자바는 `java.io.Serializable` 인터페이스 + `ObjectOutputStream`으로 객체를 바이트로 직접 변환하는 기능을 기본 제공한다. 하지만 **새 코드에서는 거의 쓰지 않는다.** Effective Java도, Oracle도 회피를 권장한다.

1. **보안 취약** — 역직렬화 시 임의 클래스의 코드가 실행되는 RCE 취약점이 반복 발견됨 (Apache Commons Collections gadget chain 등). Java 9+의 [JEP 290 Serialization Filter](https://openjdk.org/jeps/290)로 화이트리스트 제한이 가능하지만, 근본적으로 안전하지 않은 설계.
2. **JVM 종속** — 바이트 포맷이 자바 전용. 다른 언어에서 읽을 수 없어 마이크로서비스·클라이언트 통신에 부적합.
3. **버전 호환성 취약** — 필드 추가만 해도 `serialVersionUID`가 달라져 역직렬화가 깨지기 쉬움.
4. **유연성 없음** — 필드명 매핑, 포맷 커스터마이징, null 처리 등 외부 통신에 필요한 옵션 부재.
5. **크기 비효율** — 클래스 메타데이터까지 포함되어 JSON보다 큰 경우도 있음.

> Effective Java (Joshua Bloch) — Item 85: "자바 직렬화의 대안을 사용하라". 보안과 유지보수의 함정.
>
> 레거시 RMI·세션 저장 등 어쩔 수 없이 써야 한다면 **JEP 290 Serialization Filter**로 허용 클래스를 제한하는 것이 필수.

## 직렬화 동작 흐름

```
[Java 객체]
    │
    │ writeObject() / writeValueAsString()
    ▼
[직렬화 엔진] ──→ [필드 순회 + 변환] ──→ [출력 스트림]
    │                                          │
    │                                          ▼
    │                                  [바이트 / JSON / XML]
    │                                          │
    │                                          │ 역방향
    ▲                                          ▼
[Java 객체] ←── [클래스 메타 + 필드 복원] ←── [입력 스트림]
```

핵심은 **객체 그래프(참조 관계)를 어떻게 평탄화(flatten)하느냐**. JSON은 트리 구조라 순환 참조가 문제가 되고, Protocol Buffers는 미리 정의된 스키마로 처리한다.

## 실무 권장 매트릭스

| 상황 | 권장 방식 | 이유 |
|---|---|---|
| REST API | JSON (Jackson) | 표준, 가독성, 클라이언트 호환 |
| gRPC, 내부 RPC | Protocol Buffers | 작고 빠름, 스키마 강제 |
| Redis 캐시 | JSON 또는 Kryo | 디버깅(JSON) vs 성능(Kryo) |
| Kafka 메시지 | Avro / Protobuf / JSON | 스키마 진화는 Avro |
| 자바 프로세스 간 | JSON | 디버깅 쉬움, 언어 독립 |
| 자바 RMI (레거시) | Serializable + JEP 290 필터 | 다른 선택지 없음 |

> **요약**: 새 코드에서 `Serializable`은 피하고 JSON을 기본으로. 성능이 중요하면 Protobuf/Avro. 레거시 Serializable이 남아있다면 JEP 290 필터로 보안 강화.
