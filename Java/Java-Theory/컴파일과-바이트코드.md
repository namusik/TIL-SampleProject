# 컴파일과 바이트코드

> 최종 업데이트: 2026-03-27

## 컴파일 (Compile)

사람이 읽을 수 있는 소스 코드를 기계가 실행할 수 있는 형태로 **번역**하는 과정.

- 한국어 소설을 영어로 번역하는 것과 비슷함
  - 원본(소스 코드)을 읽고, 문법 오류가 없는지 확인한 뒤, 다른 언어(기계가 이해하는 형태)로 변환
  - 번역 과정에서 오타나 문법 실수가 있으면 번역가(컴파일러)가 거부 → **컴파일 에러**

### 컴파일러 (Compiler)

컴파일을 수행하는 프로그램. Java에서는 `javac`가 컴파일러.

```
.java 파일 (소스 코드)  →  javac (컴파일러)  →  .class 파일 (바이트코드)
```

### 컴파일 타임 vs 런타임

| 구분 | 컴파일 타임 | 런타임 |
|------|------------|--------|
| 언제 | 코드를 빌드할 때 | 프로그램을 실행할 때 |
| 누가 | 컴파일러 (javac) | JVM |
| 하는 일 | 문법 검사, 타입 검사, 바이트코드 생성 | 바이트코드 해석/실행, 메모리 관리 |
| 에러 예시 | 오타, 타입 불일치, 없는 메서드 호출 | NullPointerException, OutOfMemoryError |

- 식당에 비유하면: **컴파일 타임**은 주방에서 레시피를 검수하는 단계(재료 누락, 순서 오류 체크), **런타임**은 실제로 요리를 만들어 서빙하는 단계(불 조절 실패, 재료 부족은 이때 발생)

## 바이트코드 (Bytecode)

Java 컴파일러가 소스 코드를 변환한 **중간 코드**. 사람도 기계도 직접 읽기 어려운, JVM 전용 명령어 집합.

- 소스 코드와 기계어 사이의 **중간 번역본**
  - 한국어 → **에스페란토(바이트코드)** → 영어/프랑스어/일본어(각 OS의 기계어)
  - 에스페란토 통역사(JVM)만 있으면 어떤 나라에서든 소통 가능 → **Write Once, Run Anywhere**

### 왜 바로 기계어로 안 만드는가

```
C/C++:   소스 코드 → 기계어 (OS별로 따로 컴파일 필요)
Java:    소스 코드 → 바이트코드 → JVM이 기계어로 변환 (OS 상관없이 동일한 바이트코드)
```

| 방식 | 장점 | 단점 |
|------|------|------|
| 바로 기계어 (C/C++) | 실행 속도 빠름 | OS마다 다시 컴파일 필요 |
| 바이트코드 (Java) | OS 독립적, 이식성 높음 | JVM이 필요, 변환 오버헤드 |

### OS별 JVM이 기계어로 변환

바이트코드는 OS가 직접 실행할 수 없고, **각 OS용 JVM이 해당 OS의 기계어로 변환**해서 실행.

```
                        ┌→ Windows JVM → Windows 기계어
Hello.class (바이트코드) ─┼→ macOS JVM   → macOS 기계어
                        └→ Linux JVM   → Linux 기계어
```

- 바이트코드는 **하나**지만, JVM은 **OS마다 다른 버전**이 존재
- JDK를 설치할 때 OS에 맞는 버전을 받는 이유 (macOS용, Windows용, Linux용)
- "Write Once, Run Anywhere"가 가능한 이유는, OS별 차이를 **JVM이 대신 흡수**해주기 때문
- C/C++은 개발자가 OS별로 직접 컴파일해야 하는 걸, Java는 **JVM이 대신 해주는** 구조

### 바이트코드의 실행

```
.class 파일 (바이트코드)
    ↓
  JVM이 로딩
    ↓
  인터프리터가 한 줄씩 해석/실행
    ↓
  자주 실행되는 코드(핫스팟)는 JIT 컴파일러가 기계어로 변환 → 성능 향상
```

- **인터프리터** — 바이트코드를 한 줄씩 읽어서 실행. 시작은 빠르지만 반복 실행 시 느림
- **JIT 컴파일러** — 자주 호출되는 코드를 기계어로 한 번에 변환해두고 재사용. 반복 실행 시 빠름
- JVM은 인터프리터로 시작하다가, 자주 쓰이는 부분만 JIT으로 최적화하는 **혼합 방식**

## Java의 전체 실행 흐름 요약

```
Hello.java (소스 코드)
    │
    ▼  javac (컴파일러) — 문법/타입 검사 후 변환
Hello.class (바이트코드)
    │
    ▼  java (JVM 실행) — 클래스 로딩
 Class Loader
    │
    ▼  바이트코드 검증 (Bytecode Verifier)
    │
    ▼  인터프리터 + JIT 컴파일러 — 실행
    │
    ▼  OS/하드웨어
```

## 디컴파일 (Decompile)

.class(바이트코드)를 다시 .java(소스 코드)로 **역번역**하는 것.

- 번역된 영어 소설을 다시 한국어로 역번역하면 내용은 같지만 원작의 문체나 메모는 복원 안 되는 것과 같음
- 바이트코드에 로직은 남아있으므로 역번역이 가능하지만, 완벽한 복원은 아님

### 유지되는 것 vs 손실되는 것

| 유지 | 손실 |
|------|------|
| 클래스/메서드 구조 | 지역 변수명 (arg0, var1 등으로 바뀜) |
| 로직 흐름 | 주석 |
| 문자열 리터럴 | 원래 코드 스타일/포매팅 |
| 어노테이션 (RUNTIME 보존 정책) | 제네릭 일부 (Type Erasure) |

### 주요 디컴파일러

| 도구 | 특징 |
|------|------|
| IntelliJ IDEA 내장 | .class 파일 더블클릭하면 자동 디컴파일 |
| JD-GUI | 독립 실행형 GUI 디컴파일러 |
| CFR | CLI 기반, 최신 Java 문법 지원 우수 |
| Procyon | 람다, 제네릭 복원력 좋음 |

- 실무에서는 라이브러리 소스 확인이나 레거시 분석 시 사용
- 보안 관점에서, 배포된 .class 파일은 디컴파일될 수 있으므로 민감한 로직은 **난독화(Obfuscation, ProGuard 등)** 를 적용하기도 함
