# Java Memory

> 최종 업데이트: 2026-03-27 | Java 21 기준

## 개념

JVM이 프로그램 실행 중 사용하는 메모리 영역들의 구조와 동작 방식.

- 아파트에 비유하면: **공용 시설**(Method Area, Heap)은 모든 주민(스레드)이 함께 쓰고, **각 세대**(Stack, PC Register, Native Method Stack)는 주민마다 따로 있음
- 아래 각 영역은 JVM 스펙상의 **논리적 개념**이며, 실제 메모리 배치(물리적 구현)는 JVM 구현체(HotSpot 등)가 결정

```
┌─────────────────────────────────────────────────────┐
│  모든 스레드 공유                                   │
│  ┌─────────────────┐  ┌──────────────────────────┐  │
│  │  Method Area     │  │          Heap           │  │
│  │  (Metaspace)     │  │  ┌──────┬────┐ ┌──────┐ │  │
│  │                  │  │  │ Eden │ S  │ │  Old │ │  │
│  └─────────────────┘   │  └──────┴────┘ └──────┘ │  │
│                        └─────────────────────────┘  │
├─────────────────────────────────────────────────────┤
│  스레드마다 생성                                    |
│  ┌─────────┐  ┌────────────┐  ┌────────────────┐    │
│  │  Stack  │  │ PC Register│  │ Native M Stack│     │
│  └─────────┘  └────────────┘  └────────────────┘    │
└─────────────────────────────────────────────────────┘
```

## Method Area — 논리적 개념

클래스 수준의 정보를 저장하는 공간. 건물의 **설계도 보관소**와 같음 — 객체를 만들기 위한 설계 정보(클래스)가 이곳에 있음.

- 클래스 메타데이터 (클래스명, 부모 클래스, 메서드·필드 정보)
- Runtime Constant Pool (상수, 문자열 리터럴, 심볼릭 참조)
- static 변수
- JVM 시작 시 생성, 프로그램 종료 시 해제

### Method Area의 물리적 구현: PermGen → Metaspace

Method Area는 JVM 스펙상의 **논리적 개념**이고, 이 영역이 실제로 메모리 어디에 저장되느냐는 JVM 구현에 따라 다름. 이 물리적 구현이 Java 버전에 따라 변경됨.

| 구분 | PermGen (Java 7 이하) | Metaspace (Java 8+) |
|------|----------------------|---------------------|
| 위치 | JVM 힙 내부 | 네이티브 메모리 (OS 관리) |
| 크기 | 고정 크기 할당 | 동적으로 조정 |
| 문제점 | `OutOfMemoryError: PermGen space` 빈발 | 메모리 누수 시 OS 메모리까지 소진 가능 |
| 크기 제한 | `-XX:MaxPermSize` | `-XX:MaxMetaspaceSize` |

- 네이티브 메모리 = JVM 힙 외부에서 **OS가 직접 관리**하는 메모리
- 클래스 로더에 의해 로드된 클래스는 적절한 시점에 JVM이 언로드 가능

## Heap — 논리적 개념 | 물리적 구현: Young Gen + Old Gen

`new`로 생성되는 **모든 객체와 배열**이 저장되는 공간. JVM에서 가장 큰 메모리 영역이며, **GC의 주요 대상**.

- 창고에 비유하면: 새 물건(객체)은 입구 쪽 임시 구역(Young)에 두고, 오래 쓰는 물건은 안쪽 장기 보관 구역(Old)으로 옮김

### 힙 구조 (Generational)

```
┌────────────────────────┬────────────────────┐
│      Young Generation   │   Old Generation   │
│  ┌──────┬──────┬──────┐│                     │
│  │ Eden │  S0  │  S1  ││                     │
│  └──────┴──────┴──────┘│                     │
└────────────────────────┴────────────────────┘
```

| 영역 | 역할 |
|------|------|
| **Eden** | 새로 생성된 객체가 최초로 할당되는 공간 |
| **Survivor (S0/S1)** | Eden에서 Minor GC를 살아남은 객체가 이동하는 공간. S0↔S1을 번갈아 사용 |
| **Old Generation** | Young에서 여러 번 GC를 살아남은 객체 (Age 임계값 초과 시 승격) |

### 객체 생성과 메모리 할당 과정

```
1. new 키워드 → Eden 영역에 할당
2. Eden이 가득 참 → Minor GC 수행
3. 살아남은 객체 → Survivor(S0 또는 S1)로 이동, Age + 1
4. Survivor 간 이동을 반복하며 Age 증가
5. Age가 임계값 초과 → Old Generation으로 승격 (Promotion)
6. Old 영역 부족 → Major GC (Full GC) 수행
```

### 힙 메모리 할당과 해제 특성

- 힙이 한 번 확장되면, GC로 객체를 회수하더라도 **JVM은 기본적으로 메모리를 OS에 반환하지 않음**
  - JVM이 다시 메모리를 요청하는 오버헤드를 줄이기 위한 전략
  - 대량 요청 처리 후 메모리 사용량이 줄어도, 모니터링에서 메모리 사용량이 안 줄어드는 이유
- G1 GC, ZGC는 `-XX:MinHeapFreeRatio` 등을 통해 OS에 메모리를 반환하는 기능 지원
- 메모리 부족 시 → `OutOfMemoryError: Java heap space`

## Stack — 논리적 개념 | 물리적 구현: OS 스레드 스택

각 스레드마다 독립적으로 생성되는 메모리. 메서드 호출마다 **스택 프레임**이 쌓이고, 종료 시 제거됨.

- 접시 쌓기와 같음 — 마지막에 올린 접시(메서드)를 먼저 꺼냄 (LIFO)

```
Thread Stack:
┌─────────────────────┐
│ methodC() frame      │  ← 현재 실행 중
├─────────────────────┤
│ methodB() frame      │
├─────────────────────┤
│ methodA() frame      │
├─────────────────────┤
│  main() frame        │
└─────────────────────┘
```

### 스택 프레임에 저장되는 것

| 항목 | 설명 |
|------|------|
| 지역 변수 | 기본 타입 값 또는 객체 참조 (실제 객체는 힙에 존재) |
| 매개변수 | 메서드에 전달된 인자 |
| 반환 주소 | 메서드 종료 후 돌아갈 위치 |
| Operand Stack | 연산 중간 결과 저장 |

- 스택이 꽉 차면 → `StackOverflowError` (주로 무한 재귀)

### Stack vs Heap

| 구분 | Stack | Heap |
|------|-------|------|
| 저장 대상 | 기본 타입, 객체 참조 | 객체 인스턴스 |
| 생존 범위 | 메서드 실행 동안 | GC가 회수할 때까지 |
| 스레드 공유 | 스레드별 독립 | 모든 스레드 공유 |
| 속도 | 빠름 (LIFO 구조) | 상대적으로 느림 |

```java
void example() {
    int count = 10;                   // count(기본 타입) → Stack
    String name = new String("hello"); // name(참조) → Stack, String 객체 → Heap
}
// 메서드 종료 → Stack에서 count, name 제거
// Heap의 String 객체는 참조가 없으면 GC 대상
```

## PC Register — 논리적 개념 | 물리적 구현: CPU 레지스터 매핑

- 현재 실행 중인 바이트코드 명령어의 주소를 저장
- 스레드마다 독립적으로 유지 → 스레드 전환(Context Switch) 후 실행 위치 복원에 사용

## Native Method Stack — 논리적 개념 | 물리적 구현: OS 스레드 스택

- Java가 아닌 네이티브 코드(C/C++)를 JNI를 통해 호출할 때 사용하는 스택
- HotSpot에서는 Stack과 통합되어 구현되기도 함

## Garbage Collection (GC)

- Heap 영역이 GC의 주요 대상
